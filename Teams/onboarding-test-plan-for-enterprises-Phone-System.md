<a name="define-and-document-your-phone-system-with-calling-plans-in-teams-test-plan-for-enterprises"></a>定义并记录电话系统调用计划与企业团队测试计划中 
============================================================================================

已定义并记录电话系统与调用计划团队业务成功和技术实施计划中构想阶段的一部分后下, 一步验证通过满足贵公司的期望和要求功能、 功能性和易用性。 试验或最终部署在生产环境中部署之前，您应该执行此验证步骤。

您可以利用业务成功计划构想阶段，作为确定活动、 期望、 特性/功能的测试用例和总体范围的测试阶段进行评估的基础定义。

<a name="identify-testing-support-stakeholders"></a>确定测试支持利益干系人
-------------------------------------

在您准备评估具有调用计划功能的电话系统，创建测试的支持利益相关者列表，以确定支持测试阶段所需的角色。

> [!TIP]
> 随着填充测试支持利益相关者列表的团队，您可能会看到一些角色都在构想阶段，但是与倾斜向测试角色说明标识的相同。 您可能需要其他的角色，具体取决于您的测试方案的独特要求。

<a name="define-phone-system-with-calling-plans-feature-requirements"></a>与调用计划功能要求定义电话系统 
------------------------------------------------------------

作为定义调用计划范围内的用户的功能要求与电话系统的一部分，应该在构想阶段完成的第一步是[角色分析](https://docs.microsoft.com/MicrosoftTeams/phone-system-with-calling-plans#assess-environment-and-evaluate-adoption-readiness)，定义调用计划与电话系统的位置角色和方案。
标识此基线下, 一步是评估新的团队公用路线图，以确定：

-   与调用计划功能的电话系统将部署在作用域中的用户。

-   与调用计划功能的需求，提供您当前的 Skype 业务、 交换和 SharePoint 部署环境的预期的用户电话系统。

-   与调用计划功能的电话系统描述了最新的公共路线图，可以确认是否满足您的用户、 功能和范围要求在时间轴中的部署

> [!TIP]
> 最新的团队路线图为可以在上找到您的部署使用调用计划范围内的功能识别电话系统<https://aka.ms/skype2teamsroadmap>。

既然定义了与调用计划的角色和功能的电话系统，下一步评估标准将与团队的互操作性体验。 有关互操作性经验以及可用的配置选项的其他信息，请参阅[Microsoft 小组和 Skype 业务互操作性](https://docs.microsoft.com/MicrosoftTeams/teams-and-skypeforbusiness-interoperability)。

<a name="define-and-document-your-phone-system-with-calling-plans-test-plan"></a>定义并记录电话系统与调用计划测试计划
------------------------------------------------------------------

在作用域中定义了与调用计划功能的电话系统后下, 一步是创建测试计划。 从较高层次的测试计划包括整体测试策略和方法体系，您将使用在测试过程中支持的功能验证。

从较高层次的测试计划应包括：

-   **测试范围：**作为测试阶段总结重点领域 （目标、 方案和目标） 进行评估

-   **测试用例：**测试用例来验证用于与调用计划的电话系统特性在作用域中的组

-   **测试资源：**支持从环境、 技术和人员的角度来看测试工作所需的资源的一个矩阵

-   **测试计划 （时间轴）：**表示当测试将开始，多长时间很可能到最后，并且希望测试阶段结束时

-   **缺陷报告和修正：**准则应报告方式问题的测试，跟踪，和会审

-   **脱离会审和上报：**有关如何以及何时应该启动缺陷上报的大纲

-   **测试退出并挂起条件：**指导大纲的条件，可以实现注销退出测试阶段或暂停测试，直到确定了优先级的缺陷得到解决

-   **测试可交付结果：**其中开发和提供支持签收验收和退出测试过程的结果的摘要

> [!TIP]
>   测试方法可能已存在于您的组织，但下面的指南反映最佳做法可以包含也分别用于您的环境中测试团队功能。

在以下各节您会发现将帮助特定的决策和模板和主题供您选择为您完成您的测试计划中的其他规定的指导。

### <a name="define-and-document-testing-scope"></a>定义和记录测试范围

随着您着手制定测试计划，您必须定义最优先考虑，在测试范围内突出显示工作负荷，您是要评估的功能的列表。

合理评估与调用计划功能电话系统，通常的范围包括：

-   与调用计划地点准备情况的电话系统

-   电话系统调用计划用户体验

-   与调用计划管理电话系统

### <a name="define-and-document-test-defect-report-criteria"></a>定义和记录测试缺陷报告标准

如执行给定的阶段或流中的测试用例，可能会出现问题所执行测试用例的结果不是所希望的位置。

这种结果出现时，他们应呈报给指定的测试领导审查、 报告和缺陷解决方法的一个缺陷报告和修正计划中获得。

通常情况下，缺陷报告和修正计划包括以下内容：

-   **脱离 ID:**分配给问题的编号

-   **受影响的测试用例 ID**： 分配给在发现缺陷时计算该测试用例的数量

-   **缺陷描述：**问题摘要

-   **环境/重现步骤：**测试环境设置，以及重现该问题所需的确切步骤摘要

-   **严重缺陷**： 的影响的问题，范围广泛，从防止该测试用例被批准为其正在接受风险降至最低。 一些示例可能包括：

-   **低：**问题意义甚微，并且不会阻止该测试用例获得接受，如果以后再解决问题。

-   **媒体：**问题有重大影响，但不能防止该测试用例实现验收，如果测试阶段完成之前，问题可以得到解决。

-   **高：** -问题的测试用例有关键影响。 此问题必须解决，然后才能接受测试用例。

-   **状态：**已解决的问题，它是打开的还是仍在调查中

-   **提交人：**测试人员报告问题

-   **分配所有者：**从测试团队负责解决该问题分配的资源

-   **支持的详细信息：**这可以包括 — — 但不限于 — — 客户端日志、 屏幕快照或视频的问题。

测试人员执行测试用例在测试计划中所述，他们应该确保完成所出现的问题的缺陷报告和修正计划。
这将突出显示可能会阻碍或甚至暂停测试评估过程中的潜在影响。

<a name="execute-split-tunnel-enablement-validation"></a>执行拆分隧道启用验证

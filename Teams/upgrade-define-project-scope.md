# <a name="define-your-project-scope"></a>定义项目范围

![升级过程图，突出显示"Project"阶段](media/upgrade-banner-project-definition.png "升级过程的各个阶段，重点是项目定义阶段")

本文是升级过程 Project 定义阶段的一部分，创建赞助联盟后完成的活动以及您确定的利益干系人的项目团队是项目成功的关键。 在继续之前，请确认已经完成了以下活动:

- [已登记的项目利益干系人](upgrade-enlist-stakeholders.md)

花时间定义项目愿景、范围、目标和治理有助于确保所有项目利益干系人保持一致，并致力于实现相同的最终成果。 这一点尤其重要，因为技术准备团队和用户准备团队将独立协作，将各自的部分汇集在一起。 完成本部分后，请在整个项目中引用它，以确保你一切如期实现所需的结束状态。 使用下面确定的目标衡量结果，并根据需要缓解。

## <a name="project-vision"></a>Project视觉

您的愿景是"大图"或最终最终状态，它回答了"我们为什么要执行此项目？" 理想的愿景可解决组织的业务驱动因素和用户价值添加视角，如以下示例所示：

- **组织业务驱动因素**：Microsoft Teams标准化符合我们的数字工作场所转型，使我们能够提高运营效率、消除冗余解决方案并节省 500 万美元。
- **用户价值添加**：Microsoft Teams (1) 为项目笔记、文档、团队成员、对话和会议提供单个位置Office节省时间; (2) 通过使用集中式联系人列表和持久聊天跟踪来快速访问对话，简化了通信; (3) 通过在一个地方存储和访问文件来减轻查找丢失的电子邮件附件的麻烦。

请考虑以下讨论点，以帮助优化愿景：

- 当前业务流程

- 现有应用程序的挑战业务流程

- 技术如何帮助克服这些挑战

- 克服了这些挑战的情况下预期的可衡量业务成果

> [!TIP]
> 确定用例和角色，以进一步优化项目愿景。

## <a name="project-scope"></a>Project范围

你的愿景可能只会在一段时间、不同阶段中实现。 项目范围定义了此时项目的重点，有助于让项目团队专注于当前任务，从而实现长期愿景。 例如，范围可能要求运行试点、部署特定工作负荷（例如语音或会议）或启用 Teams，Skype for Business随着时间的推移规划升级。 作为项目范围的一部分，应评估：

- [各种共存模式](./teams-and-skypeforbusiness-coexistence-and-interoperability.md)， 最适合你的组织。
- 在移动到 Teams 之前，Skype for Business和Teams共存Teams。
- 是否应进行试点 [来](./pilot-essentials.md) 验证组织中技术和用户就绪性。

## <a name="project-goals"></a>Project目标

目标定义想要的结果，并使您能够衡量项目的成功。 可以将目标定义为 OKR (目标和关键) ，项目成功的衡量指标可定义为 KSIS (的关键) 。   必须让项目利益干系人全面参与定义 OKR 和 KSIS，帮助确保他们感到有所有权感，并且这些成功度量标准与定义的项目任务保持一致。 目标应包括技术和以用户为中心的成功组合。

- **OKR** 包含你在项目开始时设置的目标，以及你根据定义的节奏（例如，每月或每 (）衡量的关键) 。 通过查看关键结果，可确保项目可交付结果按计划进行，或识别和缓解问题，让项目回到正轨。OKR 通常归类为"已实现"或"未实现"。
- **KSIs** 通过详细说明好结果和/或坏结果来测量关键结果的质量与成功，并补充 OKR 的二进制性质。 定义 KSIS 时，我们建议在 SMART (标准中，使用"特定、可测量、可分配、现实、时间相关的) 条件：
  - 特定：针对特定领域进行改进
  - 可度量：量化或至少建议进度指示器
  - 可分配：指定谁将执行该任务
  - 现实：在给定可用资源后，说明可以实际实现哪些结果
  - 时间相关：指定何时可以实现结果

## <a name="risks-and-mitigation"></a>风险和缓解措施

 任何项目都可能出现不可预见的事件或其他因素，导致项目无法如期完成。必须主动评估潜在风险，并定义缓解计划来克服可能出现的问题，以便项目可以继续实现目标。 风险 _注册_ 是一个极佳的工具，用于跟踪项目风险及其可能影响及其潜在影响，以及捕获缓解计划。 下表显示了一个示例风险寄存器。

| 风险 | 可能性 | 影响 | 整体 | 缓解计划 |
|---|---|---|---|-------|
| **网络质量** | 中 | 高 | 高 | 执行网络规划练习。 |
| **低用户采用率** | 高 | 高 | 高 | 在试点和部署阶段主动与用户合作;实施有针对性的认知和培训活动，以创造期望。 |
| | | | | |

## <a name="timeline"></a>时间线

在设置升级旅程的范围时，请确保为关键里程碑设置时间线 (例如，除了完成日期之外，) 还为所有用户启用 Teams 和 Skype for Business。 定义的日程表可帮助项目团队实现一致的结束状态，并通知正确的回班日程，帮助确保项目按进度进行。请考虑一个不会太加速的日程表 (任务可能会被忽略) 或者距离过远 (其中，可能会失去) 。 理想的时间线适用于：

- **符合合规性和用户** 方案要求的产品就绪情况：请参阅产品 [](https://aka.ms/O365Roadmap)路线图，Teams为组织做好准备。
- **升级** 组：确定是按Teams组启用升级还是升级用户，这可能会影响整个升级旅程的时间线。
- **组织因素，例如更改冻结、会计年度** 结束、部署生命周期：讨论并考虑可能影响升级时间线的任何内部流程。
- **同时或** 大约同时发生的其他更改：请考虑捆绑更改或将其间距，以简化积极的用户体验并最大程度地降低对工作效率的影响。
- **资源**：与项目利益干系人确认资源分配，确保你汇集的项目团队有足够的带宽来完成所有必要的任务。

作为参考点，为升级[Pro](./upgrade-start-here.md)旅程的升级前、升级和升级后阶段提供了示例时间线，我们鼓励你根据需要进行调整，以满足组织的特定需求。

完成上述活动后，应该为项目打好基础。 继续进行技术[准备和组织](./upgrade-prepare-environment.md)[准备](./upgrade-prepare-organization.md)情况规划活动。

请记住：Skype for Business Online 将于 2021 年 7 月 31 日停用。 此后，Skype for Business不再可访问或不支持联机服务。 为了最大限度地实现效益并确保组织有适当的时间来完成升级，我们鼓励你立即开始Microsoft Teams之旅。

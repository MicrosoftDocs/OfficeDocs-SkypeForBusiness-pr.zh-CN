# <a name="update-management"></a>更新管理 
新式会议室配有Microsoft Teams 会议室设备和其他外围设备，如相机、麦克风或扬声器，以及可能更多的设备，以创建包容性和有效的会议体验。 来自不同类型的 OEM 的设备提供所需的确切组织体验：但是，必须持续使用软件和固件维护它们。  

托管服务Microsoft Teams 会议室提供保证，确保组织中的每个会议室都保持在建议级别，以提供始终准备就绪且正常工作的房间。 Microsoft 的目标是通过智能和自动化降低运营人员的复杂性和腿部工作。 将尽快执行故障排除或诊断。 

## <a name="transitioning-a-device-to-managed-services"></a>将设备转换为托管服务 
加入托管服务的会议室设备通常具有与我们的指导不同的更改管理历史记录和做法。  

- 若要从托管服务中受益，必须在托管服务组合下转换所有更新的更改管理。
- 更改管理影响事件 SLA 的多个来源，因为存在发现和修正，如果在会议室中发生事件，会重新重新启动。
- Microsoft 已实施控制和检查，以实施可能不同于一个组织和另一个组织的策略，以及在特殊情况下进行干预的能力。
- 最终，会议室设备将更新为常见标准，但因特定硬件安装问题而导致的异常除外。  

## <a name="transitioning-devices-basic-readiness-checks"></a>转换设备：基本就绪性检查 
大多数意外失败都源于基础映像中具有不确定的更改管理历史记录的更改。 

建议执行简单的就绪性检查：  

- **基础映像**：基本映像必须来自特定的 OEM。 如果设备过去已重新生成，并且在常见任务上显示意外故障或行为，则必须还原基础映像。 我们可以提供帮助，但无法远程重新生成会议室设备，因此需要本地站点技术员。  
- **基本 OS，版本：** 基本 OS 和版本必须符合Microsoft Teams 会议室设备的要求。 如果不是这样，则必须将其作为载入的一部分进行更正。 Microsoft Teams 会议室需要Semi-Annual通道服务选项下的Windows 10 IoT 企业版或Windows 10 企业版 SKU。 有关详细信息，请参阅 [地铁官方指南](rooms-lifecycle-support.md#windows-10-release-support) 。

## <a name="managed-updates--how-it-works"></a>托管更新 – 工作原理 
有两种主要方式来管理更新：  

- **自动托管**：更新基于托管服务评估安装在会议室设备中。 对于我们的项目组合中托管的更新，无需进行干预。
- **已验证环** 形：设置环形系统以预览特定设备上的更新，以便无需关联的腿部工作即可监视更新。 在广泛推出之前，环形设置提供了额外的尽职调查层。  

### <a name="automatically-managed"></a>自动托管

如果选择自动管理，则无需对部件上的更新执行任何操作。 但是，应查看托管服务支持的当前更新组合。 项目组合不断获得新的新增功能，我们的首要任务是涵盖最频繁和最具影响力的更新，以确保房间稳定。 请查看此文档) 的“更新管理”部分下的当前列表 (，规划组织所需的任何其他更改管理。  

**建议：** 请勿在任何托管设备上自行安装托管服务所涵盖的更新。 如果遇到任何问题，请在门户中报告事件。

### <a name="ring-validation"></a>环验证

选择环验证时，请查看以下部分，了解托管服务中的环的工作原理以及可用于为组织自定义环的选项。 即使进行环形验证，托管服务也尝试确保会议室不会因建议的更新而过期。 根据情况，会议室可能会收到“赶上”更新，以确保它符合托管服务建议。  

 检查门户主页和托管服务文档中的公告，因为新类型的软件和固件在产品组合中可用。 由于托管服务专家每天在我们的设备组合中审查更新版本，因此他们根据需要解决特定问题和目标更新。  

### <a name="scheduling"></a>安排 
根据会议室中的设备以及不符合适用软件和固件的托管服务标准的会议室，将计划托管更新。 

- 为了帮助客户满足更改管理要求，请在过渡圈的 **周三** 开始更新部署。 如果需要关键更新，我们将绕过此计划，并在更新可用后立即发布更新。 

- 更新根据特定会议室中的需求进行排序。 
- 如果有用于验证更新的设置环，更新将按环顺序进行。 
- 如果确定根据你的情况改善了房间稳定性，则新更新可以取代排队的更新。  
- 更新通常在夜间维护时段（本地时间 **上午 12：00 - 凌晨 5：00** ）应用，以避免任何类型的中断。 

## <a name="microsoft-teams-rooms-app-update-lifecycle-policy"></a>Microsoft Teams 会议室应用更新生命周期策略 
地铁工程团队的支持策略指出，所有支持在版本的 12 (12) 个月生命周期过期或此后已发布超过两次更新后结束。 然后，客户必须更新到受支持的版本。 请参考[Microsoft Teams 会议室应用版本支持 - Microsoft Teams |Microsoft Docs](rooms-lifecycle-support.md)详细的服务说明。

为了在所有托管会议室中保持统一的标准，并使我们能够有效地识别趋势问题，我们将根据支持和订阅服务条款和条件，支持和部署 MTR 应用软件的 N、N-1)  (N-1 最新版本或次要版本。 我们会自动使不合规的会议室保持最新状态，并根据需要绕过更新通道。 

N-1 策略也适用于第三方软件。

### <a name="updates"></a>更新  

此视图显示租户的相关更新及其各自的状态。 若要查看不再处于活动状态的过去更新，请选择“ **包括过去更新** ”切换到 ON。  

任何更新都可以处于以下状态之一：

| 地位 | 说明 |
|:- |:- |
| 计划 | 为给定环中的会议室计划更新。 请记住，只有在进度到达会议室所在的环后，更新才会显示“已计划”。 例如，如果过渡圈中存在新的更新，则它只会显示“临时”圈中会议室的计划。<br></br><p> 在更新进入该环之前，其他环将具有“不需要”状态。</p> |
| 正在进行中 | 正在进行更新，单个环显示状态。 此状态显示整体环形状态，因此，如果更新适用于租户中过渡圈中的单个房间，则更新将处于“正在进行”状态，直到达到执行环。 |
| 已完成并失败 | 更新已完成所有已配置的环的进度，并且至少在一个会议室中失败。 |
| 完成 | 更新已完成所有已配置的环的进度，并已成功安装在所有适用的会议室中。|
| 废弃 | 更新已停用。 进一步部署已停止。 这是典型的，因为更新已被新版本取代。 |
| 暂停 | 更新处于暂停状态。 |
| 不需要 | 更新要么尚未评估房间，要么不适用于会议室。 |

### <a name="rings"></a>环  

环用于降低从功能更新的部署派生的问题的风险。 这是通过逐步将更新部署到整个站点来完成的。 每个环应有一个Microsoft Teams会议室的列表和相应的推出计划。 定义环通常是一次性事件 (或至少不常) ，但 IT 应不时重新访问这些组，以确保顺序仍然正确。  

“ **指环** ”选项卡列出了租户中的所有环。 有三个预配置的环：  

- **过渡**：将会议室分配到过渡圈，这是你的测试版。 所有新更新将首先在此处推出。 一般情况下，需要确保过渡圈表示环境中设备类型的多样性的会议室。 如果某些类型的会议室具有不常见的配置或出现问题的历史记录，请考虑在过渡中代表它们。

- **常规**：默认情况下，所有房间都放置在此环中。 企业中使用的大多数会议室设备属于此类别。 

- **执行人员**：此组应包括你最高调的会议室，你希望在其中主动最大程度地减少中断。 一个很好的示例是用于执行会议或大型团队会议的大型会议室。 

### <a name="specifying-rollout-timeline"></a>指定推出时间线

更新不能超过 60 天才能跨所有环完成。  

|参数 |解释 |
|:- |:- |
|延迟期|更新从第一个环开始后，延迟期是在此环上启动更新之前的几天内的延迟。|
|推出持续时间|<p>在此环上开始更新后，此圈中将部署此圈。 例如，如果持续时间为 5 天，则更新在此环上启动后，它将部署在此环中的会议室 5 天以上。|
|测试期|一旦应用到环中，用于测试/验证环中的更新的天数。 测试期在推出完成后开始，一旦完成，更新将移动到下一个环。|
|完成时间|“完成时间”列指示此圈完成 (推出持续时间 + 测试期) 的总天数。|
|总时间|底部是“总计”行，指示从第一个环到最后一个环完成更新需要多长时间。|

## <a name="managed-updates-visibility-and-control"></a>托管更新：可见性和控制

托管服务在整个组织中协调更新。 但是，你可以根据需要进行可见性和控制。 下面是以下方法： 

- 如果更新失败，则会自动与 Microsoft 托管服务运营团队一起生成票证。 运营团队将采取措施修正故障，并在必要时与你联系。  
- 如果看到导致问题的更新，可以使用 **“暂停** ”按钮暂停更新。 按下“暂停”按钮会为操作中心创建一个要调查的票证。 请务必在暂停更新时提供详细信息，以帮助加快事件响应。  
- 如果在会议室中看到更新失败，并且已更正网络断开连接等合理原因，则可以使用 **“重试所有失败** ”按钮重试更新。  
- 当您可能决定提前提供更新时，可能会出现紧急情况。 在这种情况下，可以使用 **“强制更新** ”按钮。 使用“强制更新”选项时，可以选择立即强制更新，或在下一个会议室可用时强制更新。  

> [!NOTE]
> **我们不建议将“强制更新”** 作为常规更新管理策略。 如果推送仍在验证传递中的更新，可能会遇到我们已经了解的问题。 在这种情况下，将尽最大努力解决此类会议室的事件。  

- 此外，为了确保良好的更改管理做法，我们将在服务内部记录每种力量更新。 将来，我们期望你也能够看到这一点。

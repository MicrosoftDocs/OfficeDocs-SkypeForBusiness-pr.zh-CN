# <a name="steering-committee"></a>调控委员会

## <a name="steering-committee-overview"></a>调控委员会概述 （英文)

调控委员会是主要利益干系人和已恢复的项目负责人调控小组一起用于指导项目或向其定义的业务产出的程序。 调控委员会不直接负责*如何*传递项目，但而不显示*哪些*项目实现的业务。

每个项目需要商定的构想和责任。 若要提供的项目的所需的产出构想必须明确定义，它需要监视和维护。 这将成为调控委员会的责任： 驱动器决策、 建议、 提供战略监督用作拥护到项目的计划，组织者和 — 在必要时 — 删除窗口阻止程序。

您的组织应对大量缜密调控委员会组建。 委员会必须确保项目实现业务目标为驱动更改整个组织满足定期来讨论项目，当前脉冲已定义和帮助删除过程中遇到任何障碍旅程。

委员会应定义其责任是提供包括一些关键目标：

-   保留项目工作组和执行发起人或 executive 高领导层之间强对齐方式。

-   提供到执行发起人或 executive 领导的状态的项目的见解。

-   允许执行发起人或 executive 领导团队提供指导和输入到项目并确保它与总体业务目标，通过调整项目计划、 目标关键结果 (OKRs) 和其他项目活动对齐。

调控委员会满足的时间间隔定期整个生存期内的项目来确保组织领导和项目团队之间的对齐方式。 此关键会议确保项目的方向具有高领导层的完全支持并包含任何反馈提供高领导层到驱动器成功项目。 委员会使用这些会议深入了解到项目状态，以及为：

-   达成一致对齐到业务案例，并确保项目将朝传递这些成果的业务产出。

-   检查并批准项目准确性和合规性业务案例。

-   检查并验证与业务案例可能会影响任何定义的产出所做的更改。

-   进行有关项目可交付结果的优先顺序的战略决策并批准临时可交付结果。

-   标识、 管理和缓解差距、 风险和问题其中其他影响是需要从委员会。

-   收集从执行发起人或管理层的领导团队需要升级，确定优先级和解决任何冲突利益干系人业务部门之间的问题的支持。 

-   提供正式反馈和 executive 领导、 更改顾问委员会，或其他业务和 IT 利益干系人，如果适用的建议。

## <a name="identify-members-of-the-steering-committee"></a>确定调控委员会的成员

一个委员会才有效其参与者都有影响力。 调控委员会需要维护来自业务和技术决策者而言，以及任何核心项目工作组领导和合作伙伴团队参与解决方案的表示形式。

每个调控委员会需要强领导和明确定义的角色和职责。 若要设置委员会的性能和其成员的参与现实的预期值，必须文档并同意分配的角色和职责。 绝对必须响应执行发起人以及业务领导基于此委员会以跨组织提供有效的影响，并确保多样化的角度。

> [!NOTE]
> 调整角色和职责基于您的组织和项目工作组设计。 在较小的组织，这些角色可以相互重叠。

## <a name="schedule-steering-committee-meetings"></a>安排调控委员会会议 

尽管每个组织和每个项目工作组在结构略有不同的方式，调控委员会会议决不是另一个项目短会议。 请考虑调控委员会会议的目标： 它是您通过收集支持和指导执行发起人或 executive 领导在跟踪保留项目的机会。 请确保所有人参加此会议，以便没有直接整个项目工作组和执行发起人或 executive 高领导层之间消息并没有任何错误传达中继邮件时。

安排会议委员会时, 请考虑的项目和组织节奏。 尝试以安排每周或每隔一周，取决于您执行发起人的首选项和您的组织的变化。

> [!TIP]
> 如果您是快速移动环境，不要让会议的时间间隔太长时间 —，可能会导致无法通过更频繁地满足避免的问题。 但是，在速度较慢控制进度的环境中，按住会议每周可能会导致会议疲劳，可能会降低调控委员会的有效性。 使用您的体验和判断确定适当的节奏，并始终记住，您可以更改计划。 不满足只是为了会议。

### <a name="hold-the-meeting"></a>召开会议 

会议本身应自然延续核心团队所执行的准备工作。 请记住这些会议的目标是来培养的通信和协作可以获得该项目的愿景氛围。 使用以下建议以帮助确保成功的会议：

1.  安排会议要面对面。 使用一个表，足以每个人 sit 标识会议室。 确保会议室内具有视频功能。 尽管它采用理想可使每个人聊天室，但它并不总是可能，因此请确保会议邀请包括会议室和联机会议信息与至少音频 — 但理想情况下，音频、 视频和共享。 鼓励使用视频，当用户加入会议时的所有远程参与者。

2.  尝试以保留操作的议程，尽管无疑将出现分散注意力，需要解决。 如果可能 （具体取决于其重要性） 将这些项目添加到"驻留区域"部分，并解决这些更高版本。 如果推迟太重要主题，则获取协议继续讨论，但实现需要将一些其他议程项目移动将来加入会议。 同意留出一些时间 （5 到 10 分钟） 来解决需要立即关注委员会的任何按项目。

3.  准备讨论任何打开的支持情况。 尝试将讨论尽可能此会议，高级 — 票证数量、 票证，主要打开项目的类型。 您应做好解决所有问题，即使它们是次要，了解对的业务和项目的效果的影响

4.  委派便笺格式。 很难演示时，或中间直热争议做笔记。 请确保记录委员会中所做的任何决定的详细信息。 准备发送这些决策和 notes 出的后续的电子邮件的一部分。

5.  始终尝试升级有关人的工作工作量和用户反馈的正整数新闻 （如果可用）。 告知和销售案例。 如果完全负反馈，请准备讨论该产品，并解释哪些措施来解决负用户体验问题。

    a.  需要为 kudos 给各个团队成员提供的良好他们所做的工作的机会。 务必在工作组成员，尤其是前面其领导团队完成出色的、 硬工作时识别。

6.  始终保存会议达成一致任何后续步骤并确定在会议期间达成这些操作项目所有者的末尾。

7.  向与会者发送后续电子邮件，其中包括您创建，演示文稿，并将追加所做的任何注释。 请确保包含未参与的任何人。

## <a name="next-steps"></a>后续步骤

复查调控委员会的概述，现在就操作。项目团队应开始到项目的时间线实施调控委员会和准备正则调控委员会会议节奏其工作。 理想情况下，将形成委员会之前项目开始和指导各个阶段，但不能情况下，如果窗体委员会指导到成功结果的项目工作组。

1.  确定调控委员会成员。 问自己：

    -   我们有来自业务和高领导层的右表示吗？

    -   是我们标识的成员能够影响其业务部门或以其他单位表示的对等方？

2.  创建通信将发送到调控委员会的每个成员：

    -   包含角色表。

    -   销售委员会的重要性。

3.  设置第一个会议讨论角色和职责、 定义委员会的责任和验证或定义 OKRs。 也可以包括议程项目：

    -   讨论会议节奏。

    -   验证责任和 OKRs 驱动向有意义的业务产出。

    -   确认策略与 OKRs 对齐。

4.  启动每个对话中驱动质量：

    -   详述无质量冠军的重要性。

    -   标识的人员或团队人员将承担质量冠军角色。

调控委员会成功不发生通过好运。 选择正确的成员，提供最佳的意图，可以仍产生较差的结果。 为了帮助驱动器的调控委员会成功，您需要确保您具备以下：

-   商定的构想和范围将提倡委员会的定义完善实实在在的业务产出项目。

-   驱动器故意会议良好的议程。

-   开放式交流的氛围。

    -   打开通信和协作将授权委员会。 创建位置都能听到所有的语音和见解会考虑的环境。 独特的观点可以具有的最大的影响。

    -   当领导操作以提升成员协作和通信时，可以启用开放环境。 下面是一些准则：

        -   所有委员会成员都应要引入到组织更改强拥护者。

        -   保留对那些绝对需要提供准确和那些可以使或跨业务部门影响决策委员会大小限制。 窗口阻止程序出现时，了解在组织的层次结构和每个成员的直接影响行将至关重要。

        -   委员会讨论和想法中从事的项目经理和团队主管。 准备好要邀请其他核心团队成员清楚置于特定区域中可能需要的详细讨论。

在解决方案的生命周期将更改委员会的角色。 构想和入职培训移动到的驱动器值阶段给委员会可能需要几个调整：

-   调整会议的节奏。 查看组织的动态并确定是否需要较少发生会议。 为组织更改开始 dwindle 将可能会降低上级通信的频率。 调查更改为月份或季度会议节奏必要的会议

-   调整会议中的代表。 当解决方案移动到的驱动器值阶段很可能的核心项目工作组，将重新分配给新项目。 将所需更新报告为支持团队和操作将成为上的解决方案状态的潜在顾客。

-   定义和报告操作 OKRs 针对。 关键结果和成功指标将随解决方案的生命周期。 所需的业务产出应继续跟踪和报告。 继续报告的质量和可靠性达到业务产出驱动的上下文中的语音解决方案。 其他指标和密钥结果，例如用户的观点，用户应用和支持指标将成为主要的成功指标。

-   使用较大的调控委员会总括性涵盖了的多个解决方案的驱动器值阶段。 查找更有效地使用调控委员会的机会。 使用委员会跨解决方案更大球形提供指导。

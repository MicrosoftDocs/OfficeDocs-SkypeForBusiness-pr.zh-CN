# <a name="steering-committee"></a>指导委员会

## <a name="steering-committee-overview"></a>指导委员会概述

指导委员会是由关键利益干系人与项目负责人共同管理小组，这些利益干系人与项目负责人共同指导项目或计划实现定义的业务成果。 指导委员会不直接负责项目的交付方式，而负责项目向业务交付什么。 

每个项目都需要达成一致的愿景和规定。 若要实现项目的所需成果，必须明确定义愿景，需要监视和维护该愿景。 这将成为指导委员会的责任：推动决策、提出建议、提供战略监督、作为组织的项目计划的支持者，并在必要时删除阻止者。

组织应大量考虑组建指导委员会。 委员会必须确保项目实现你为推动整个组织的变化而定义的业务目标，定期开会讨论项目的当前发展情况，并帮助消除在过程中遇到的任何障碍。

委员会应定义其规定，以包括一些关键目标：

-   在项目团队与执行发起人或高层领导之间保持强大的一致性。

-   向执行发起人或高层领导提供有关项目状态的见解。

-   通过调整项目计划、目标关键结果 (OKR) 和其他项目活动，允许执行发起人或高层领导团队为项目提供方向和意见，并确保它与总体业务目标保持一致。

指导委员会在整个项目生命周期内定期开会，以确保组织领导与项目团队保持一致。 此关键会议可确保项目方向获得领导层的完全支持，并将领导层提供的任何反馈纳入到项目中，以推动成功。 委员会使用这些会议来深入了解项目状态，以及：

-   就符合业务案例的业务成果达成一致，并确保项目推动这些成果的交付。

-   检查和批准项目，确保项目的准确性和合规性符合业务案例。

-   查看并验证对业务案例所做的更改，这些更改可能会影响任何已定义的结果。

-   就项目可交付结果的优先级做出战略决策，并审批临时可交付结果。

-   确定、管理和缓解需要委员会施加其他影响的差距、风险和问题。

-   针对需要升级、确定优先级和解决利益干系人业务部门之间任何冲突的问题，从执行发起人或高层领导团队获得支持。 

-   向管理层、变更顾问委员会或其他业务和 IT 利益干系人（如果适用）提供正式反馈和建议。

## <a name="identify-members-of-the-steering-committee"></a>识别指导委员会的成员

委员会只有在参与者具有影响力时有效。 指导委员会需要保持业务和技术决策者以及参与解决方案的任何核心项目团队领导和合作伙伴团队的代表性。

每个指导委员会都需要强大的领导和明确定义的角色和责任。 若要为委员会的表现及其成员的参与设定现实的期望，必须记录并同意分配的角色和责任。 您必须在此委员会中具有响应迅速的高层发起人和业务领导才能在整个组织中提供有效影响，并确保从不同角度出发。

> [!NOTE]
> 根据组织和项目团队设计调整角色和责任。 在小型组织中，这些角色可以重叠。

| 标题 | 角色 | 联系人 | 电子邮件 |
|----------------------------|------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|---------|-----------|
| 执行发起人 | 项目推广者，确保其方向与组织的愿景保持一致。 充当委员会无法同意的决策的最终授权。 | | |
| Business Decision Maker ()  | 项目提倡者，确保其方向与组织的愿景保持一致 | | |

| 核心Project团队 | 提供有关任何特定组件的详细信息，包括技术或非技术 (，例如采用和更改管理)  | | |
| 质量冠军 | 负责将质量指标放在一起。 | | |

## <a name="schedule-steering-committee-meetings"></a>安排指导委员会会议 

尽管每个组织和每个项目团队的结构略有不同，但指导委员会会议绝不应成为另一个项目代表会议。 考虑指导委员会会议的目标：通过从执行发起人或高层领导收集支持与指导，让项目保持进度。 确保每个人都参加此会议，以便整个项目团队与执行发起人或高层领导之间直接沟通，在中继消息时不会传达任何错误的消息。

安排委员会会议时，应考虑项目和组织的节奏。 根据执行发起人首选项和组织的速度，尝试每周或每隔一周安排一次。

> [!TIP]
> 如果你的环境是一个快速移动的环境，请不要让会议之间的时间过长，这可能会导致问题，这些问题可能通过更频繁的会议来避免。 但是，在速度较慢的环境中，每周举行一次会议可能会导致会议疲劳，这会降低指导委员会的效率。 使用你的体验和努力来确定适当的节奏，并始终记住你可以更改日程安排。 不要仅仅为了会议而开会。

### <a name="prepare-for-the-meeting"></a>准备会议

若要充分利用每个人的时间，你需要为会议做准备。 核心团队可以开始为指导委员会会议准备材料。 您可以创建一个PowerPoint，以帮助准备会议。

应完成以下资产，并包含在会议演示文稿幻灯片中：

-   跟进以前的指导委员会操作项目

-   指导委员会PowerPoint，项目高级状态报告

    -   从每月/每季度/每年类型评审中查看项目。

    -   确定关键项目里程碑：

        -   讨论里程碑的重要性和影响 (应在风险注册记录中捕获) 。

        -   根据准备工作确定即将进行的迁移，尤其是业务决策者的迁移。

    -   指出项目执行日期和影响较高的活动。

    -   报告被确定为对委员会重要的 OKR;请务必包括使用情况、用户情绪、质量和可靠性指标。

-   指导委员会PowerPoint，项目每周状态报告

    -   包含突出显示和低光、关键问题、操作项目以及下一步。

    -   包含更多侧重于日常问题的技术详细信息，而不包含有关总体项目状态的技术详细信息。

-   问题日志 [和风险注册](1-envision-define-my-success-cloud-voice.md#define-okrs-ksis-and-risks)

    -   共享委员会正在跟踪的任何问题的状态。

        -   准备好提供任何和所有 bug 的更新。

    -   确定可能影响成功实现业务成果的任何新项。

        -   确定它们对项目的影响。

        -   确定受影响的个人或业务单位。

        -   确定下一步，以及可能需要从委员会成员获得哪些帮助。

-   质量概述：使用为用户准备[网络Teams。](prepare-network.md)

    -   准备深入解释这些质量报告的结果，并包含修正所发现问题的建议。

    -   每周调整质量报告，以重点关注以下方面：

        -   组织的整体质量趋势

        -   已知问题网站或遇到问题的站点的状态，用于显示网站的方向，以及（理想情况下）报告修正工作的正面结果

        -   基于项目进度的最近启用的网站的质量状态

接下来，安排一次呼叫，为指导委员会会议做准备。 此调用应仅包括核心项目团队的关键团队成员，这可能包括来自网络、桌面、运营或其他团队的资源，这些资源具有将在委员会会议期间处理的项目。 核心项目团队成员永远不应受到你在指导委员会呼叫期间计划涵盖的任何材料的警惕。

这种准备性呼叫是在指导委员会开会之前进行同步的一种好方法。 此外，还需要确保项目团队中任何未处理的项目、不一致或不一致事件都得到公开。 这有助于促进开放沟通，防止任何人在会议期间感到意外。

与核心团队和合作伙伴利益干系人一起查看该周的报告，并解决任何问题。 在团队内根据讨论调整、添加或删除任何内容，或同意向指导委员会请求进一步澄清。 分配将展示每个分区的团队成员，以帮助确保流程和连续性。

### <a name="hold-the-meeting"></a>召开会议 

会议本身应该是核心团队所做的准备工作的自然延续。 请记住，这些会议的目标是促进沟通和协作的环境，以实现项目的愿景。 使用以下建议来帮助确保会议成功：

1.  安排面对面会议。 确定一个会议室，其表格足够大，每个人都可以坐在一起。 确保会议室具有视频功能。 虽然让每个人都在会议室是理想选择，但并非始终可行，因此请确保会议邀请包括会议室和联机会议信息（至少包含音频，但理想情况下是音频、视频和共享）。 鼓励所有远程参与者在加入会议时使用视频。

2.  尝试保持日程，但毫无疑问，需要解决的干扰。 如果可能 (其重要性) ，请将它们添加到"停车场"部分，稍后再处理它们。 如果主题非常重要，无法推迟，请同意继续讨论，但认识到需要将其他一些日程项目移到将来的会议中。 同意留出一些时间 (五到十) ，以解决任何需要委员会立即注意的紧急项目。

3.  准备好讨论任何打开的支持案例。 尽量使此会议的讨论保持高级别 - 门票数量、票证类型、主要未处理项目。 你应该准备好解决所有问题，即使它们很小，也请了解对业务的影响和对项目的影响

4.  委派记笔记者。 当你正在进行展示或进行一场自如的讨论时，很难做笔记。 确保记录在委员会中做出的任何决策的详细信息。 准备作为后续电子邮件的一部分发送这些决定和注释。

5.  如果可用，请始终尝试宣传有关人员工作努力和用户反馈的 (积极) 。 讲述并销售故事。 如果反馈是完全负面的，请准备好进行讨论，并解释解决负面用户体验问题所执行的工作。

    a.  请利用机会为各个团队成员提供 kudos，以完成他们正在完成的工作。 必须识别团队成员何时完成出色而困难的工作，尤其是在领导团队的面前。

6.  始终保存会议结束，以就任何下一步达成一致，并确定在会议期间达成一致的操作项目的所有者。

7.  向与会者发送包含您创建的演示文稿的跟进电子邮件，并追加任何笔记。 请务必包括未参加的任何人。

## <a name="next-steps"></a>下一步

查看指导委员会概述后，现在可以采取行动了。项目团队应开始工作，将指导委员会实施到项目日程表中，并准备定期指导委员会会议节奏。 理想情况下，委员会将在项目开始之前组建，并在其所有阶段提供指导，但如果情况并非如此，请组建委员会来指导项目团队取得成功的结果。

1.  确定指导委员会的成员。 询问自己：

    -   我们是否具有来自业务和领导层的合适代表形式？

    -   我们确定的成员能否影响其他单位中的业务部门或对等单位？

2.  创建要发送给指导委员会的每个成员的通信：

    -   包括角色表。

    -   销售委员会的重要性。

3.  设置第一次会议，讨论角色和责任，定义委员会规定，并验证或定义 OKR。 日程项目还可以包括：

    -   讨论会议节奏。

    -   验证该规定和 OKR 是否正在推动有意义的业务成果。

    -   验证策略是否与 OKR 保持一致。

4.  开始提高每个对话的驾驶质量：

    -   阐明拥有质量冠军的重要性。

    -   确定将担任"质量冠军"角色的人或团队。

指导委员会成功不是因为幸运而发生的。 选择具有最佳意图的合适成员仍会产生较差的结果。 若要帮助推动指导委员会的成功，需要确保具有以下各项：

-   具有明确定义且可衡量的业务成果的项目的一致愿景和范围，委员会将支持这些成果。

-   组织有序的日程，推动有目的的会议。

-   开放通信的环境。

    -   开放式沟通和协作将赋予委员会支持。 创建一个能够听取所有声音并尊重意见的环境。 唯一视角可以产生最大的影响。

    -   当领导层行动促进成员协作和通信时，可以启用开放环境。 下面是一些指南：

        -   所有委员会成员都应是组织中引入的变更的强提倡者。

        -   将委员会规模限制为绝对需要提供准确图片的人，以及能够跨业务部门做出或影响决策的人。 当出现阻止程序时，了解组织的层次结构和每个成员的直接影响线至关重要。

        -   让项目经理和团队领导参与委员会讨论和工作。 准备好邀请其他核心团队成员，清楚说明特定领域可能需要的深入讨论。

在解决方案的生命周期内，委员会的角色会发生变化。 随着构想和载入进入驱动器价值阶段，可能需要对委员会进行一些调整：

-   调整会议节奏。 查看组织的动态，并确定会议是否需不经常召开。 随着组织变化开始减少，高级通信的频率可能会降低。 调查在适当时将会议更改为每月或每季度会议节奏

-   调整会议中的代表。 随着解决方案进入驱动器价值阶段，核心项目团队可能会重新分配到新项目。 需要更新报告为支持的团队，并且操作将成为解决方案状态的潜在顾客。

-   定义和报告操作 OKR。 关键结果和成功指标将在解决方案的生命周期内发生变化。 应继续跟踪并报告所需的业务成果。 在推动业务成果的情况下，继续报告语音解决方案的质量和可靠性。 其他指标和关键结果（例如用户情绪、用户采用和支持指标）将成为成功的主要指标。

-   使用更大的指导委员会保护套涵盖多个解决方案的驱动器价值阶段。 寻找更有效地使用指导委员会的机会。 使用委员会提供有关更大范围解决方案的指导。

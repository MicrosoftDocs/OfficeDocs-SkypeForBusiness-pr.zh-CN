# <a name="plan-for-local-media-optimization-for-direct-routing"></a>规划直接路由的本地媒体优化

公用电话交换网 (PSTN) 语音被视为业务关键型应用程序，对语音质量寄予厚望。 直接路由允许你控制媒体流量流，以适应全球各种企业的大量网络拓扑和本地电话设置。

直接路由的本地媒体优化使你可以通过以下方式管理语音质量：

- 控制媒体流量如何在 Teams 客户端与客户会话边界控制器之间流动， (SBC) 。
- 使媒体在公司网络子网的边界内保持本地。
- 允许在 Teams 客户端和 SBC 之间传输媒体流，即使 SBC 位于具有专用 IP 的企业防火墙后面且不直接对 Microsoft 可见也是如此。

本地媒体优化支持两种方案：

- 通过连接到主会话启动协议的集中式 SBC 集中所有本地中继 (SIP) 中继-为公司的所有本地分支机构提供电话服务。

- 构建 SBC 的虚拟网络拓扑，其中本地分支机构中的 SBC 连接到集中代理 SBC，该代理 SBC 通过其外部 IP 地址可见并与之通信。 在虚拟网络拓扑中，下游 SBC 通过内部 IP 进行通信，对电话系统不直接可见。

本文介绍功能以及客户方案和解决方案。 有关配置的详细信息，请参阅 [配置本地媒体优化](direct-routing-media-optimization-configure.md)。

  > [!NOTE]
  > 如果要在 Intranet 的边界内保留本地媒体，建议进行本地媒体优化。 如果已有媒体旁路，并且仅使用 SBC 的公共 IP 地址，则不强制迁移到本地媒体优化。 可以继续使用媒体旁路。 有关详细信息，请参阅 [计划媒体旁路](direct-routing-plan-media-bypass.md)。

有关哪些 SBC 供应商支持本地媒体优化的信息，请参阅经 [直接路由认证的会话边界控制器](direct-routing-border-controllers.md)。

## <a name="supported-customer-scenarios"></a>支持的客户方案

对于此讨论，假设 Contoso 在全球运行多个业务，如下所示。  (请注意，欧洲和亚太地区仅用作示例。 一家公司可能有多个要求相似的不同区域。) 

- **在欧洲**，Contoso 在大约 30 个国家/地区设有办事处。 每个办公室都有自己的专用交换机 (PBX) 。

  Contoso 被提供了一个选项，用于将所有 30 个欧洲办事处的中继集中在一个位置-阿姆斯特丹。 Contoso 在阿姆斯特丹部署了 SBC，提供了足够的带宽来通过集中位置运行呼叫，将中央 SIP 中继连接到集中式位置，并开始为阿姆斯特丹的所有欧洲位置提供服务。

- **在亚太地区**，Contoso 在不同国家/地区拥有多个办事处。

  在许多国家，该公司仍然在本地分支机构 (TDM) 中继进行时分复用。 在 APAC 区域中，TDM 中继的集中化不是一个选项，因此无法切换到 SIP。 假设亚太地区有 50 多个 Contoso 分支机构， (SBC) 数百个网关。 在此方案中，由于缺少公共 IP 地址和/或本地 Internet 突破，无法将所有网关与直接路由接口配对。 此外，有些国家/地区实施法规要求，如果没有本地 PSTN 网络连接，就无法满足这些要求。

Contoso 根据其业务要求，使用本地媒体优化实现两种解决方案，用于直接路由：

- **在欧洲**，所有中继都是集中式的，并且根据用户位置在中央 SBC 和用户之间流动媒体。

  - 如果用户连接到公司网络的本地子网 (即用户是内部) ，则媒体在中央 SBC 的内部 IP 和用户的 Teams 客户端之间流动。

  - 如果用户在公司网络边界之外（例如，如果用户使用公共无线 Internet 连接），则该用户被视为外部网络。 在这种情况下，媒体在中央 SBC 的外部 IP 和 Teams 客户端之间流动。

- **在 APAC 区域中**，集中式代理 SBC 与 Microsoft 直接路由配对，后者在直接路由接口和本地分支机构的下游 SBC 之间定向媒体。

  本地分支机构中的下游 SBC 对 APAC 中的直接路由不直接可见，但它们通过使用 Set-CSOnlinePSTNGateway cmdlet 在 Microsoft 电话系统中创建虚拟网络拓扑进行配对。 如果可能，媒体始终保留在本地。 外部用户具有在 Teams 客户端与代理 SBC 的公共 IP 之间流动的媒体。

## <a name="local-media-optimization-modes"></a>本地媒体优化模式

本地媒体优化支持两种模式：

- **模式 1：始终绕过**。 在这种情况下，如果用户是内部用户，则媒体将流经本地下游 SBC 的内部 IP 地址，而不管内部用户的实际位置如何;例如，在下游 SBC 所在的同一分支机构内或位于其他某个分支机构中。

- **模式 2：仅适用于本地用户**。 在此模式下，仅当内部用户与下游 SBC 位于同一分支机构时，媒体才会直接流向本地下游 SBC 的内部 IP 地址。

若要区分本地媒体优化模式，租户管理员需要使用 Set-CSonlinePSTNGateway cmdlet 将每个 SBC 的 -BypassMode 参数设置为“Always”或“OnlyForLocalUsers”。 有关详细信息，请参阅 [配置本地媒体优化](direct-routing-media-optimization-configure.md)。

> [!NOTE]
> 如果用户是内部用户， **则需要** 通过内部 IP 地址在用户与 SBC 之间建立媒体连接。 在这种情况下，没有媒体的公共交通中继回退，因为 SBC 将为媒体连接提供内部 IP。

### <a name="mode-1-always-bypass"></a>模式 1：始终绕过

如果分支机构之间有良好的连接，则建议的模式是“始终绕过”。

例如，假设一家公司在阿姆斯特丹有一个集中式 SIP 中继，该中继为 30 个国家/地区提供服务，并且所有 30 个站点和本地用户之间都有良好的连接。 德国还有一个分支机构，部署了本地 SBC。

德国的 SBC 可以配置为“始终绕过”模式。 用户，无论其位置如何，都将通过 SBC 的内部 IP 地址直接连接到 SBC (例如从法国到德国：有关参考) ，请参阅下图。

### <a name="mode-2-only-for-local-users"></a>模式 2：仅适用于本地用户

如果本地分支机构之间的连接不正确，但每个本地分支机构和区域办事处之间的连接良好，则建议的模式为“仅适用于本地用户”。

例如，在 APAC 区域中，假设 Contoso 在不同国家/地区有多个办事处。 对于许多国家/地区，无法切换到 SIP，因为该公司在许多本地分支机构中仍有 TDM 中继。 在 APAC 区域中，TDM 中继的集中化不是一个选项。 此外，亚太地区有 50 多个 Contoso 分支机构，有数百个网关 (SBC) 。

若要构建一个解决方案，即在 TDM 中继集中化不是选项的 APAC 区域的所有本地分支机构中提供 PSTN 服务，Contoso 管理员将新加坡的一个区域 SBC 配对为直接路由服务的代理 SBC。 当地分支机构之间的直接连接并不好，但每个当地分支机构与新加坡区域SBC之间有着良好的联系。 对于区域 SBC，管理员选择“始终绕过”模式，对于本地下游 SBC，管理员选择“仅针对本地用户”模式。

下面描述了两种方案：

- 方案 1. 用户与联机语音路由策略中定义的 SBC 位于同一位置

- 方案 2. 用户和网关位于不同的站点中

#### <a name="scenario-1-the-user-is-in-the-same-location-as-the-sbc-defined-in-online-voice-routing-policy"></a>方案 1. 用户与联机语音路由策略中定义的 SBC 位于同一位置

### <a name="what-is-the-cqd"></a>CQD 是什么？

使用调用质量仪表板 (CQD) 为深入的业务服务通过团队和 Skype 电话的质量。 CQD 旨在帮助 Skype 业务和团队管理员和网络工程师优化网络。 CQD 探讨聚合信息，为整个组织整体模式可能会明显，允许人员进行明智的呼叫质量评估。 CQD 提供让您深入了解总体的通话质量、 电话可靠性和用户体验的调用指标的报告。

> [!NOTE]
> CQD 不包含任何个人身份信息 (PII)。 PII 是可用于单独或与其他信息来标识、 联系或找到一个人，或以识别上下文中的个体的信息。

#### <a name="audio-pcr-"></a>音频的 PCR % 

音频差调用比率 (PCR) 表示调用具有音频质量差的单位的总体百分比。 此统计数据是为了突出显示的区域，您的组织可以专注努力产生强大影响朝着减小此值，并改善用户体验，这就是托管的网络时观察 PCR 是主要原因。 外部用户是很重要，但调查不同组织和用户的基础上。
考虑提供最佳做法对于外部用户，并查看独立于整个组织的外部调用。

#### <a name="call-setup-failures-"></a>调用安装程序失败百分比 

这表示无法建立任何媒体会话。 此处测量用户体验上，给出影响的严重性，目的是降低此值改为接近零。 此度量值高是更比成熟的部署，在新部署中不完整的防火墙规则与常见但仍然很重要，要定期进行监视。 随着您操作的严格程度的逐渐成熟，您可以展开该度量包括视频和桌面共享工作负载。

#### <a name="call-drop-failures-"></a>调用放失败百分比 

这适用于会话意外终止音频工作负荷。 随着您操作的严格程度的逐渐成熟，您可以展开该度量包括视频和桌面共享工作负载。

#### <a name="client-and-device-readiness"></a>客户端和设备的准备工作

您需要一个稳定的客户端和设备战略，以确保您的用户具有一致和良好的用户体验。 几个关键原则推动每个准备工作策略。

##### <a name="client-readiness"></a>客户端准备工作

强大的客户端准备工作策略可确保您的用户正在享受最佳的体验同时运行最新版本的客户端。
Microsoft 的例行修补 Skype 业务客户端;确保将其最新的环境中对您整体的成功至关重要。

我们建议您不让落后超过六个月由您的客户端版本。 如果您正在使用 Office 单击运行，您就已经正在保持最新服务。 使用包含的[客户机报告](#determine-client-versions)，在本指南的后面部分所述来帮助您完成此过程。 您还可以利用速率我调用示例报告，以进一步提高您的客户端准备工作战略。

> [!IMPORTANT]
> 目前，分布式团队客户端和 Azure 内容传递网络通过自动更新并将保留最新的服务。 客户端准备工作和调查活动并不适用于小组。


##### <a name="device-readiness"></a>设备就绪

没有一种单一策略会影响用户体验比您设备的准备工作战略。 大多数组织很高兴地从用户 （例如，桌面电话或其他专用音频设备） 中删除不必要的设备，这通常是用于切换到团队或 Skype 为业务核心业务合理性论证。 然而，这些相同的组织有时不愿意再提供替换的设备，即使这些设备成本较低。 现代笔记本电脑和 Pc，但配有内置的麦克风和扬声器，不适合企业级语音 ip (VoIP)。 通常如果这会产生不好的体验所有参与者，特别是演讲者是很嘈杂的环境中。 Microsoft 的设备认证计划确保用户使用任何设备认证团队或 Skype 业务参与电话呼叫，它产生了优于未经认证的设备体验。

我们始终建议团队和业务用户的 Skype 使用认证的耳机或扬声器在参与语音呼叫使用桌面客户端。
有关 Microsoft 认证的设备的详细信息，请查看此[文章有关电话和经鉴定的设备](https://technet.microsoft.com/office/dn788944.aspx)。 [设备报告](#devices-investigations)，以后在本指南中，用于管理您的设备的帮助。 速度我调用示例报表也可用于进一步提高您的设备准备工作战略。

### <a name="user-experience"></a>用户体验

分析了用户体验比科学更多艺术，因为这里收集的指标不总是意味着没有网络或服务问题，但相反，它们表明用户可以体验问题。 Microsoft 提供了一种内置的调查机制--已知为率我调用 (RMC) — — 以帮助衡量整体用户体验。 RMC 将帮助您回答下面的问题，从用户的角度来看：

-   我知道如何使用解决方案吗？

-   易于使用且直观，是解决方案，它支持我日常通信的需要？

-   该解决方案帮助我完成工作了吗？

-   我总体感觉是解决方案的什么？

-   可以使用解决方案在某一点的时间，而不考虑我的位置？

-   可以设置并维护一个调用？

#### <a name="rmc"></a>RMC

RMC 建团队和业务的 Skype，并被自动配置为显示后每隔 10 个电话或 10%的所有调用中的一个。 这份简短的调查询问用户评价调用并提供很少的上下文为何呼叫质量可能做得很差。 一个或两个评级被认为较差、 三到四属正常，而五个是优秀。 尽管有些绝热指示符，这是服务度量可以错过的发现问题的一个有用指标。

> [!NOTE]
> 直到用户将被要求通过通常授予除响应不正确，您的意见对 RMC 调查响应回来作为排山倒海般负面。 大多数用户只响应时通话质量较差。 因此，可能出现 RMC 报告偏态差侧服务指标都很好时，即使。 


您可以使用 CQD RMC 用户响应，报告和 CQD 模板中包括的示例报告。 不过，它们不在本指南中详细讨论。 教育用户提供有用的 RMC 响应的指南和关于 Skype 的在线业务中的 RMC 的详细信息，请参阅此[博客文章](https://blogs.technet.microsoft.com/jenstr/2015/05/05/rate-my-call-in-skype-for-business-2015/)。

#### <a name="service-management-tasks"></a>服务管理任务

在云第一个世界中，必须执行某些服务管理任务，以维护高质量的用户体验。 这些任务范围从确保足够的带宽不饱和互联网链接，验证该服务质量 (QoS) 是在地方上所有受管理的网络区域，到达服务以及 — — 最后 — — 掌握[Office 365 IP 范围上防火墙](https://support.office.com/article/Office-365-URLs-and-IP-address-ranges-8548a211-3fe7-47cb-abb1-355ea5aa88a2)。

#### <a name="network-tasks"></a>网络任务

有两种类别的网络任务： 可靠性和质量。 可靠性的重点测量用户的能力，成功地进行调用并保持联系。 质量重点发给团队和 Skype 的在线业务过程的用户的客户端和调用结束后的聚合遥测数据。

给出可靠性具有对用户体验的关键影响，开始评估和深入质量之前研究这些指标。

#### <a name="endpoints-tasks"></a>终结点任务

此类别中的主要任务验证哪些客户端版本上从过去的六个月，以确保用户得到的好处对 Skype 业务桌面客户端所做的持续优化桌面生成运行业务的 Skype。 此外，这简化了整体的客户端管理任务，并提供一致的用户体验。

其他重要的领域是监视的设备在您的部署中普遍和推动经过认证的设备，以提供最佳用户体验。

### <a name="first-vs-second"></a>第一次与第二个 

许多维度和度量值中 CQD 被划分为第一或第二。
CQD 不使用调用方/被调用方字段 — — 因为有调用方和被调用方之间的中间步骤，这些已被重命名为_第一次_和_第二_。 以下逻辑用于确定将流或通话中涉及的哪些终结点标为第一。

-   第一次将始终 （会议服务器、 中介服务器等） 的服务器终结点服务器涉及到流或调用。

-   其次将始终为客户端终结点除非流在两个服务器终结点之间。

-   如果两个终结点具有相同的类型，这些选择是第一个基于内部订购的用户代理类别。 这样可以确保排序的一致性。

在这两个相同时确定的第一个或第二个终结点的详细信息，请参阅[维度和度量值中 CQD 可用](https://aka.ms/cqd-dm)。

### <a name="stream-vs-call"></a>与呼叫流

您需要了解正确选择的维度或度量值，您将会看到中的 CQD 的调用和流之间的区别。

**流：**只有两个终结点之间存在一个流。 没有为每个方向，只有一个流和两个流所需的通信。 流可以用于分析建筑物或网络。 在某些情况下，调用和流的名称 （例如，调用安装程序流或调用下沉流） 中使用。
这些还都属于单个流。

**调用：**调用是从所有参与者的所有流的分组。 调用组成 — — 至少 — — 两个流。 一次调用中将有两个参与者每个最小的一个流。 调用可以用于分析趋势，随着时间的推移。

到调用或流引用维度或度量值的其他指南，请参阅[维度和度量值中 CQD 可用](https://aka.ms/cqd-dm)

### <a name="good-poor-and-unclassified-calls"></a>良好、 较差，以及未分类电话

调用被归类为良好、 较差，或者未分类。 让我们花点时间来谈谈更多详细信息中的每个。

**很好或很差：**好或差的调用包含，包含一套完整的服务标准，为生成完整的 QoE 报告了一个调用。
确定调用是否好或差是介绍[本指南中前面](#pcr)。

**未分类：**未分类的电话中没有一套完整的服务标准。 这些通常是简短的电话 — — 通常小于 60 秒 — — 其中无法计算平均值，QoE 报告没有生成。

#### <a name="supernetting"></a>Supernetting

您可以使用 supernetting，通常称为无类别域间路由 (CIDR，) 定义各个子网的位置。 *Supernet*是组合的几种共享单个路由前缀的子网。 而不是添加一个条目为每个子网，您可以使用 supernetted/CIDR 地址。 Supernetting 支持，但我们不建议使用它。

例如，Contoso 的营销建筑组成下面的子网：

-   10.1.0.0/24--第一层

-   10.1.1.0/24 – 二楼

-   10.1.2.0/24--第三个平面布置

-   10.1.3.0/24--第四的地面

您可以添加一个条目为每个子网，而不是使用 supernetted/CIDR 地址 — — 在本示例中，10.1.0.0/22。

-   网络 = 10.1.0.0

-   网络范围 = 22

以下是几点要实现 supernetting 之前，请考虑：

-   Supernetting 采用较少的时间提前，但代价是降低数据的丰富程度方面。 让我们假设没有质量问题涉及网 200.1.2.0。 如果实现 supernetting，不知道在建筑物中子网的位置或哪种类型的网络 （例如，实验室）。 如同建筑物的所有子网定义并上载楼层位置信息，您可以查看该区别。

-   请务必确保 supernetted/CIDR 地址是正确的并不捕捉不需要子网。

-   Supernetting 可以使用 8 位为 28 位掩码生成映射中。

-   它是非常常见，在数据中查找 192.168.0.0。 对于许多公司来说，这表示该用户是在家里。 对于其他人，这是 IP 地址方案为附属机构。 如果您的组织具有使用此配置的办公室，不包括其在构建文件中很难区分家庭和内部网络，通过使用通用的子网。

> [!IMPORTANT]
> 用于表示 supernet 网络范围。 所有新生成的数据文件上载将检查有任何重叠范围。 如果您以前上载的构建文件，应下载当前文件并上载再次来识别任何重叠并修复该问题。 任何重叠以前上载的文件的情况可能会导致错误的子网与报告中的建筑映射。

### <a name="devices-investigations"></a>设备调查

若要使使用的下列设备报告，它的最大努力理解这一概念的平均意见成绩 (MOS)。 MOS 是黄金标准量化指标，以衡量感觉的音频质量。 它表示为从 0 到 5 的整数分级。

所有的语音质量的措施的基础是语音的一个人如何感觉质量。 因为它人类感知的影响，它是本质上具有主观性。 有几种不同的方法进行主观测试。
大多数的语音质量度量基于绝对分类 (ACR) 评级。

在 ACR 主观测试中，人数统计方面的重大评级及其质量的经验范围为 1 （坏） 到 5 （极好）。 分数的平均值是 MOS。 结果 MOS 取决于经验至组和类型的体验被评定为公开的范围。

因为它是不实际执行实时通信系统的语音质量的主观测试，团队和业务的 Skype 使用高级的算法来客观地预测主观测试的结果生成 MOS 的值。

可用组 MOS 以及相关衡量标准提供视图到传递给用户体验的质量。

通过提供与设备认证团队和业务的 Skype 的用户，您可以减少遇到由于设备本身 （这是更有可能，例如，对于便携式计算机的内置扬声器和麦克风） 的负面经验的可能性。 有关详细信息，请参阅[电话和 Skype 的业务的设备](https://technet.microsoft.com/office/dn947482)。

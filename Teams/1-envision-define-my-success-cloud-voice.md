# <a name="define-my-success"></a>定义我成功

这篇文章概述用于定义您的组织部署的音频会议、 与调用计划，或电话系统直接路由的电话系统的成功的要求。 通过正确定义哪些成功如下所示，您可以通过部署为您进行测量结果，并验证您获得的结果是，应该怎么做的。

<!--ENDOFSECTION-->

**音频会议**提供通过允许会议参与者使用传统的固定，专用电话进行拨入加入通过公用电话交换网 (PSTN) 的其他入口点 （临时或计划） 的任何会议的组织交换机 (PBX) 电话或移动电话。 这很有用，组织者或参与者时不在计算机前或数据连接时不可用或太可靠，以支持语音通信 — 例如不移动数据范围远程区域或连接到可用的公共 Wi-fi服务带宽有限或会议参与者希望拨入会议，通过使用电话服务终结点的轻松访问它们。

**与调用计划 （"调用计划"） 的电话系统**为组织提供了一种方法通过使用户可以从其计算机和移动设备发起与业务相关电话呼叫改进其工作区。 工作区现代化可以是任意数量的方案的一部分 — 基于活动的工作实现，主要 office 移动，传统的 PBX 解决方案、 PSTN 服务提供程序合同，依此类推组成部分停用 office 调整出刷新。 与调用计划，Microsoft 方便了连接到 PSTN。

**电话系统直接路由 （"直接路由"）** 为相同的好处的调用计划，上面列出的组织提供了，之处在于通过第三方提供程序，而不是 Microsoft 方便了 PSTN 连接。 这样可以部署在国家/地区其中调用计划不可用，或在部署中现有 PSTN 服务提供程序合同需要进行维护或与特定的本地系统的互操作性所需的位置。 一个其他方案，应考虑直接路由是电话系统的互操作性。 时用户正在转换到团队中调用，某些用户可能会保留在旧式 Pbx 上。 直接路由启用这两个用例的共存。 组织内保持之间的旧式系统上的用户和团队用户的呼叫流量。

<!--ENDOFSECTION-->

## <a name="define-business-use-cases-for-audio-conferencing-calling-plans-or-direct-routing"></a>为音频会议、 调用计划，或直接路由定义业务用例

首先核心项目利益干系人需要定义支持音频会议、 调用计划，或直接路由中实现的业务使用案例。

业务用例，旨在定义和文档预期和实实在在的业务产出，并包括以下各项：

-   当前的业务流程的说明

-   使用现有的业务流程的挑战

-   技术如何帮助克服这些挑战

-   克服了这些挑战的情况下预期的可衡量业务成果

> [!TIP]
> 以下是音频会议已完成的业务用例示例：
>|         |
>|---------|
>|**当前业务流程说明**<br>Contoso 当前依赖现任本地电话服务提供商提供的 PSTN 会议服务，按内部会议和涉及外部各方的会议的会议分钟数收费。|
>|**现有业务流程上的挑战**<br>Contoso 大约等待 USD1 100 万每年的当前的 PSTN 会议服务的内部会议所产生的成本的 75%。 传统电话加入会议的 PSTN 会议服务由承载终结点使用未与组织为现代的通信和协作平台采用团队规划对齐。|
>|**技术如何能够克服这些挑战**<br>采用 Microsoft 团队作为现代的通信和协作平台，内部用户将主要使用其 Pc 配有优化的耳麦和会议室内设备加入会议。 音频会议服务将在支持外部参与者，或以支持使用 PC 音频不是为内部参与者理想的情况下提供。|
>|**预期的可衡量业务成果**<br>向工作组移动作为现代的通信和协作平台，再加上要进行音频会议服务将大幅降低成本提供 PSTN 会议服务。|

## <a name="define-okrs-ksis-and-risks"></a>定义 OKRs、 KSIs 和风险

使用组合项目利益干系人，您可以将业务使用案例、 组织范围和项目日程表转换目标和关键结果 (OKRs) 和项目成功的度量值可定义为关键成功指示器 (KSIs) 的列表。

在定义 OKRs 和 KSIs 项目利益干系人的完全参与至关重要，以帮助确保他们认为一种所有权以及对齐成功事件记录到组织的业务要求这些措施。

OKRs 包含您的项目，开头中设置的目标，并且每季度定义可测量关键的结果。 查看每月关键结果来跟踪的整个项目，状态和 — 基于进度 — 根据需要调整季度的计划。

KSIs 测量质量和成功的关键结果，并通过详细报告良好和/或错误结果补充 OKRs （达到或未实现） 的二进制特性。

在定义 KSIs 时，我们建议您使用"特定实实在在、 赋值、 实际、 与时间有关"（智能） 的条件：

-   特定于： 面向特定区域的改进

-   可测量： 量化或至少提供建议的进度指示器

-   谁将执行此操作可分配： 指定

-   写实： 状态结果可以真实实现给定可用资源

-   与时间有关： 指定可以时获得的结果

## <a name="establish-a-steering-committee"></a>建立调控委员会

调控委员会是主要利益干系人和已恢复的项目负责人调控小组一起用于指导项目或向其定义的业务产出的程序。 调控委员会不直接负责*如何*传递项目，但而不显示*哪些*项目实现的业务。

每个项目需要商定的构想和责任。 提供您想从项目的结果，构想必须明确定义，并且它需要监视和维护。 这将成为调控委员会的责任： 驱动器决策、 建议、 提供战略监督用作拥护到项目的计划，组织者和 — 在必要时 — 删除窗口阻止程序。

您的组织应对大量缜密调控委员会组建。 委员会必须确保项目实现业务目标为驱动更改整个组织满足定期来讨论项目，当前脉冲已定义和帮助取消阻止沿遇到任何障碍方式。

委员会应定义其责任是提供包括一些关键目标：

-   保留项目工作组和执行发起人或 executive 高领导层之间强对齐方式。

-   提供到执行发起人或 executive 领导的状态的项目的见解。

-   允许执行发起人或 executive 领导团队提供指导和输入到项目并确保它与总体业务目标，通过调整项目计划、 目标关键结果 (OKRs) 和其他项目活动对齐。

调控委员会满足的时间间隔定期整个生存期内的项目来确保组织领导和项目团队之间的对齐方式。 此关键会议确保项目的方向具有高领导层的完全支持并包含任何反馈提供高领导层到驱动器成功项目。 委员会使用这些会议深入了解到项目状态和：

-   达成一致对齐到业务案例，并确保项目将朝传递这些成果的业务产出。

-   检查并批准项目准确性和合规性业务案例。

-   检查并验证与业务案例可能会影响任何定义的产出所做的更改。

-   进行有关项目可交付结果的优先顺序的战略决策并批准临时可交付结果。

-   标识、 管理和缓解差距、 风险和问题其中其他影响是需要从委员会。

-   收集从执行发起人或管理层的领导团队需要升级，确定优先级和解决任何冲突利益干系人业务部门之间的问题的支持。 

-   提供正式反馈和 executive 领导、 更改顾问委员会，或其他业务和 IT 利益干系人，如果适用的建议。

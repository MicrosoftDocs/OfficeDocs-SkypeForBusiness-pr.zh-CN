# <a name="change-management-in-lync-server-2013"></a>Lync Server 2013 中的更改管理

</div>

<div id="mainSection">

<div id="mainBody">

<span> </span>

_**上次修改的主题：** 2014-08-18_

对 IT 环境的更改是不可避免的。 更改包括新技术、系统、应用程序、硬件、工具、过程和角色和职责方面的更改。 有效的更改管理系统允许您快速引入 IT 环境的更改，并减少服务中断。 变更管理系统将更改系统所涉及的团队汇集在一起。 例如，决定利用 Office Web Apps。 这是一个集成的 Lync Service 应用程序，使用户能够在浏览器中读取和编辑文档。 此服务的实现在投入生产后，需要多个团队的参与：

  - **测试团队**   此团队加载-在测试服务器上测试 Office Web Apps，该过程提供有关生产服务器的预期使用模式和预期性能的信息。

  - **Lync 管理员**   此团队确定部署策略并在可能的情况下将安装脚本。 团队负责确保更改已部署在生产环境中，并且负责在以后进行管理。 团队必须了解更改的影响，并在将更改投入生产之前将其纳入过程中

  - **网络团队**   此团队负责对防火墙规则的更改，这些规则允许从 Internet 访问内部 Lync 池服务器。 团队还负责与 Lync 管理员合作，以确保可用带宽能够支持额外负载。

  - **安全团队**   此团队将评估安全性并将风险降至最低。 安全团队必须查看已知的漏洞并帮助确保最大限度地降低安全风险。

  - **用户验收团队**   此团队由愿意测试系统并提供改进反馈的用户组成。

变更管理过程定义每个团队的责任，并安排要执行的工作，并在需要时纳入检查和测试。 更改控件将因更改的复杂性和预期效果而异。 它们可能因次要更改的自动审批、将评审会议更改为完整项目级评审而异。 为了更好地说明这一点，本节将对更改组进行讨论。

  - **主要更改**   主要更改对系统有全局影响，可能需要来自不同团队的输入。 升级到 Lync Server 2013 的一个示例。 主要更改会影响许多团队和可能不同的系统。 更改管理过程可能包含一个或多个变更评审会议，以通知团队将参与更改或受更改影响。

  - **重大更改**   重大更改需要大量的资源来进行规划、构建和实施。 应引入相应的更改控件，以帮助确保能够理解更改的效果，测试部署过程，并且回滚和应变计划已准备就绪。 一个重大更改的示例是部署新的累积更新。

  - **次要更改**   微小更改不会显著影响 IT 环境，例如，通过 Microsoft Lync Server 2013 控制面板更改某些 Lync 策略。

  - **标准更改**   ：定期执行标准更改，并对其进行充分了解和记录。 更改管理过程应检查对过程所做的所有更改。 与创建内容数据库或添加用户相比，常规更改不需要使用它。

下面的更改管理示例检查不同的团队如何交互以及在部署新 service pack 时执行的操作。 这些操作通过更改管理过程进行组织和管理。

  - **提出更改请求**   安全团队已评估了最新的 service pack，并确认它可解决生产系统中可能存在的漏洞。 团队引发更改请求，以将新的累积更新应用于运行 Lync Server 的所有服务器。

  - **Service pack 发行说明回顾**   ： Lync 管理员团队审阅 service pack 发行说明，以确定对系统的影响。

  - **执行一系列实验室测试时**   ，Lync 管理员团队必须在测试环境中的服务器上执行测试更新，以确定是否可以成功应用 service pack，而不会影响任何已安装的应用程序和服务器系统。 如果在生产环境中有与 Lync Server 交互的第三方或内部创建的应用程序，则还应对这些应用程序进行测试。 这些测试还可用于估计执行升级所需的时间。

  - **通知用户发生中断时**   ，Lync 管理员团队、通信团队或用户帮助台会通知所有受影响的用户有关计划的维护周期以及服务将不可用的时间。

  - **在升级之前执行 lync 的完整备份**   。在升级之前，lync 管理员团队必须验证是否存在可用于还原为原始系统状态的有效备份（如果 service pack 安装失败）。 建议将备份还原到备用服务器，以使此系统在出现问题时易于使用。

  - **部署累积更新时**   ，Lync 管理员团队会在计划的维护周期内执行安装。

## <a name="managing-the-timing-of-changes"></a>管理更改的计时

我们建议您实施计划更改过程以避免工作的重叠部分中断。 例如，两个团队可能都在规划对系统的微小更改。 一个团队可能会在另一个团队将旧版用户迁移到该池中时对池应用累积更新。 其他团队正在规划的更改不会影响任何团队，并且每个团队可能都不一定知道其他团队正在规划的更改。 如果同时发生这两个更改，则可能会在实施更改时遇到问题。 此外，如果在应用更改后存在问题，例如，如果用户迁移失败，可能很难决定应回滚的更改。 应在 IT 和管理之间设置定期维护期间，以测试更改并接受这些更改。

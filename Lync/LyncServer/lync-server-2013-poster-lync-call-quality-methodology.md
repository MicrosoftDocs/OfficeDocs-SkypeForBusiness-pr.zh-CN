## <a name="what-is-lync-cqm"></a>什么是 Lync CQM？

通话质量方法是一种新的疑难解答和服务管理框架，可更好地集中精力改进 Lync 中的企业语音服务。 使用 CQM 时，需要付出较少的精力才能确保呼叫质量和用户对企业语音服务的满意度。 [通话质量方法](http://go.microsoft.com/fwlink/p/?linkid=615208)中的 CQM 更充分地解释。 本文和海报是该内容的摘要。

CQM 将系统故障排除分解为三个路径或 "道路"。 这些是：服务器植物的道路，它看起来是服务器和它们之间的链接、端点道路，它查看用户设备和用于执行呼叫的媒体，以及用于解决传统交换电话网络通话的集成的最后一英里路上。

每个道路都划分为与特定区域或主题相关的多个段，并且在每个段定义中可以获得哪些内容是可接受的质量级别，采取行动以达到该质量级别，并实施服务管理计划以保持该质量级别，然后再转到下一主题。

标牌将 Lync CQM 作为一局游戏提供给三位玩家，每个玩家都将经历其中一条道路。 下载中包含的卡用于模拟必须克服的障碍通话质量。 有关目标以及如何实现它们的提示和建议包括在三个路径中，以及确定首次在实际应用程序中首次着手的优先级原则（在游戏中，所有三条道路均以并行方式解决）。

CQM 在早期版本的 Lync 中如何工作？ CQM 是 Lync 2013 的新增功能，但其中大部分功能可用于 Lync 2010。 CQM 可能会对 Microsoft Office Communicator 有一定的作用，但此操作未经测试且不受支持。

如果对如何使用 CQM 有疑问，可以将问题提交到 cqmfeedback@microsoft.com。

## <a name="prioritize-run-trending-queries"></a>优先顺序：运行趋势查询

CQM 的第一步是运行两周的每个趋势查询，然后分析结果。 按最大流参与者、最高不良流比率和托管区域（你控制的区域）优先顺序纠正操作。如果音频/视频多点控制单元（AV MCU）或中介查询显示较差的结果，请从红色或服务器植物路上开始。如果有线或无线查询显示较差的结果，请从蓝色或最后一英里路上开始。如果 VPN 或外部查询显示较差的结果，请从绿色或结束点路上开始。

选择开始使用的道路后，为每个区域定义一个目标（Assert），工作以满足该目标（实现），然后实施过程来保持目标（维护）。 您还可以将此海报用作游戏，以了解 CQM 背后的原则。

## <a name="managedunmanaged"></a>托管/非托管

Lync 服务器部署和网络基础结构通常可分为托管和非托管空间。 托管空间包括您的整个内部有线网络和服务器基础结构。 非托管空间是无线基础结构和外部网络基础结构。

这种区别提高了数据的清晰度，并帮助您的组织集中关注将对用户的语音和视频质量产生显著影响的工作负荷。 如果将呼叫放置在你拥有（托管）的基础结构上，而不是部分受其他实体（非托管）控制的基础结构上，则用户有不同的预期质量。 这并不是说无线用户离开其自己的设备，以便获得出色的 Lync 服务器体验。

在非托管空间中提高语音质量要求在托管空间中具有高质量。 无线（Wi-fi）是否被视为托管或非托管空间由你的组织决定。 实现正常环境的技术在两个空间中有所不同，就像解决方案一样。

## <a name="the-server-plant-road"></a>服务器植物道路

服务器植物公路的第1部分用于解决 Lync 实现中的实际服务器。 收集有关服务器本身及其在实现中的角色的 KHI 数据并分析结果。 如果操作是保证的，请更正发现的任何问题。 有关此主题的更多详细信息，请在 KHI 海报附带的[Lync Server 2013 中介绍的关键运行状况指示器](lync-server-2013-poster-key-health-indicators.md)中介绍。

下一段用于解决 AV MCU 服务器和中介服务器之间的媒体流。 首先确定数据流阈值较差的目标。 较差的流通常\>是 PacketLossRate .01 \>或 PacketLossRateMax。 另一个理想目标是\< PoorStreamsRatio 2%。 接下来，使用详细查询查找具有不良流的 AVMCU 和中介服务器对，调查不良流的原因，查看不良流路径中的网络设备，纠正不良流，并为网络定义最佳或 "黄金" 配置机房. 若要保持成就，请实施流程和工具以管理配置偏移，并报告新的问题区域。

接下来，检查中介服务器和公共交换电话网络（PSTN）网关之间的媒体流。 首先确定数据流阈值较差的目标。 较差的流通常\>是 PacketLossRate .01 \>或 PacketLossRateMax。 另一个理想目标是\< PoorStreamsRatio 2%。 接下来，使用详细查询查找具有不良流的中介服务器和网关对，调查不良流的原因，查看不良流路径中的网络设备，纠正不良流，并为网络定义最佳或 "黄金" 配置机房. 若要保持成就，请实施流程和工具以管理配置偏移，并报告新的问题区域。

最后，检查你的 PSTN 网关的运行状况指标。 标识显示运行状况并针对它们定义目标的统计信息。 此处未提供任何特定指南，因为可以使用多个可能的网关。 建立目标后，根据需要进行补救以实现目标;在该过程中，你可能会为网关定义 "金色" 或最佳配置。 若要保持成就，请实施流程和工具以管理配置偏移，并报告新的问题区域。 请注意，固件和软件更新可能会改变你的配置，或导致你更改 "黄金" 配置的定义，因此请注意这些活动。

## <a name="the-last-mile-road"></a>最后一英里的道路

客户端连接到网络的两种方法中，有线预期提供最高的质量，并且相应地必须是最后一个英里问题的初始焦点。 使用 CQM 有线查询（LastMile\_0\_有线）和它提供的不良流比率数据。 我们建议为包含\< \> 300 流的网站定义目标 PoorStreamsRatio 5%）。 若要实现目标，请将从最差订购的子网修正为最佳，并实施 QoS。

优化有线连接的质量后，提高无线质量将变得更容易，因为无线基础结构位于每个位置的有线内核顶部。 具有良好有线质量的网站中的较差无线流必须属于特定无线组件。 CQM 无线查询（LastMile\_1\_无线）在某个日期范围内运行，并将你环境中的所有内部无线流从 Lync 客户端返回到任何会议服务器或中介服务器。 我们建议为包含\< \> 300 流的网站定义目标 PoorStreamsRatio 5%）。 若要实现目标，请将从最差订购的子网修正为最佳，并实施 QoS。

## <a name="the-end-points-road"></a>终结点道路

在与 Lync 配合使用时，通过耳机和已知的其他设备生成可接受质量，开始查询。 我们建议使用超过100个\>流的实现目标 AvgSendListen MOS 3.6。）通过识别有问题的设备并修复或替换它们来实现目标。

接下来，检查设备或电脑处理音频以查找最终用户呼叫。 建议的目标质量指标为 AudioMicGlitchRate \<= 1。 为用户系统确定最佳系统配置时，请定义一个 "黄金" PC 配置，包括驱动程序版本。

现在，检查音频流从 Lync 终结点系统获取的网络路径，这可能会导致音频质量较差。 如果音频通过 VPN 连接，您可能会看到延迟问题。 如果内部 Lync 客户端无法为两方或对等调用的其他内部 Lync 客户端建立直接媒体流，则它将回退到通过 Lync Edge 服务器中继的路径，从而导致延迟问题，并增加损失和抖动。 我们建议你通过 VPN 定义质量指标0% 的媒体。 在你更正以实现你设置的目标时，请确定问题子网并调查防火墙规则、数据包 shapers 和其他相关网络设备配置。

IP 数据包可以使用 "传输控制协议（TCP）" 或 "用户数据报协议（UDP）"。 TCP 对于数据流而言是最佳的。 UDP 是无连接的，并且更高效地适用于媒体，因为 TCP 恢复机制无法以实时媒体的损失解决。 Lync 始终首选 UDP，但如果无法建立 UDP 会话，则将还原为 TCP。 通过 TCP 的媒体会话将表现出比通过 UDP 更差的质量。 我们建议通过 TCP 的0% 连接的质量定义。 在你更正以实现你设置的目标时，请确定问题子网并调查防火墙规则、数据包 shapers 和其他相关网络设备配置。

## <a name="service-management"></a>服务管理

服务管理是 CQM 的结束状态，所有三条道路的目的地。 若要保持高级别的通话质量，请监视以下方面：

1.  **用户**-补救活动应显示用户满意度的显著增加。 你可以通过问题票证或其他反馈机制来衡量此问题。 您还可以发布质量指标。

2.  **流程**-定义 operationalize CQM 的每天、每周和每月流程。 在你进行补救时（每天），监视节奏将以更高的频率启动，并随着你的稳定而移动到较低频率（每月）。

3.  **工具**-确定用于测量和修正的工具。 你可能会发现自动运行 CQM 查询以支持你的进程非常有用。 补救措施可能需要其他工具，例如，在网络元素上强制实施标准化配置或在不良流中排除损失。

## <a name="board-game-rules"></a>董事会游戏规则

你可以将此海报用作对 CQM 实现的引用或用作练习概念的游戏。 要玩游戏，您需要 1 6 面片和所提供的卡。 可在标准 Avery 5871 名片上打印可下载版本的卡。

游戏适用于3个玩家。 玩家可以使用三条路径来达到所需质量并达到中心服务管理状态：服务器植物、终点和上一英里。 每个路径在断言质量目标、实现目标和维护系统方面的方式上都有停止。 将卡放在上面的指定区域，然后绘制5张卡片。 查看你绘制的卡片并将其放在相关的版块段上。 每个玩家通过其路线中的卡片逐个移动，断言质量目标，实现这些目标，以及维护服务级别。 当所有玩家都达到中心服务管理状态时，游戏就完成了。 游戏卡下载中提供了更详细的规则。

若要**断言**质量目标，请查看适用于该目标的参数，并向您发送的内容提供相关内容，并不会选择接受。 我们推荐了起始点，但您必须进行最后的通话。 此异常是 KHI 数据，应使用 Microsoft 建立的标准。 请参阅随附的 KHI 海报。

若要在游戏中**实现**，请使用提供的卡来代替 KHI 数据和系统查询。 如果在游戏开始时未绘制与给定方位有关的卡，则可以继续使用它。 如果有相关卡，请滚动骰子。 如果您在卡上显示的数字下滚动，则您已成功。 如果你在指定的数字上滚动，则必须从该幻灯片中绘制另一个卡片。 如果该卡指示两个或更多玩家需要滚动，则它们必须全部成功滚动。

要在游戏中**维护**，请将有关 Lync 环境的服务管理计划更高的状态。

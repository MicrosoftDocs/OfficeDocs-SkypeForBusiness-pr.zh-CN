# <a name="plan-for-monitoring-in-skype-for-business-server-2015"></a>在 Skype for Business Server 2015 中规划监视
 
**摘要：**同时规划在 Skype 的监视服务业务服务器 2015年仔细阅读本主题。
  
在 Skype 的监视服务的业务服务器 2015年提供一种方法使管理员能够在其组织，使他们可以识别趋势和问题进行通信会话的收集使用情况和质量数据。 持续监视您的部署可以及早发现方面的问题并保持组织的用户满意。 
  
监视在 Skype 的业务服务器不需要单独的服务器角色 （如是这种情况在早期的 Lync 版本）;相反，每个前端服务器内置监视服务。 监视是服务器未启用默认情况下，在 Skype 业务。 这篇文章将帮助您确定是否启用监视，期间或之后您初始 Skype 业务服务器的配置，并且您需要哪些 SQL 资源支持监视活动。 如果你不是很确定哪些内容被监控或哪些内容不被监控，以及监控有什么帮助，请参阅[监控基础知识](monitoring.md#Basics)。 要开始执行规划过程，请参阅[定义监控需求](monitoring.md#requirements)。 有关监控的 SQL 要求的详细信息，请参阅[监控的 SQL 要求](monitoring.md#topologies)。
  
## <a name="basics-about-monitoring"></a>监控基础知识
<a name="Basics"> </a>

会话是用户连接到答： 通用术语
  
- 会议
    
- 会议工具，如音频/视频或应用程序共享
    
- 通过点对点对话（如即时消息或音频呼叫）的其他用户
    
> [!NOTE]
> Skype 业务服务器跟踪的每个会话的信息： 谁名谁;在会话中; 使用了哪些终结点持续多长时间未在会话;感觉到; 质量是会话的什么等等等等。 不，Skype 业务服务器不会记录和存储实际调用本身。 其中包括即时消息会话： 尽管 Skype 业务服务器记录即时消息会话的有关信息，它不会保留在会话期间发送的每条即时消息的记录。 
  
基本呼叫详细信息收集到的信息通过 Skype 业务服务器为每个会话可以用于：
  
- **投资回报率 (ROI)** 分析。 管理员可以进行比较类似数据以显示成本节约并帮助论证 Skype 业务服务器的部署及其以前的电话系统为收集使用情况的数据。
    
- **设备库存管理**。资产管理信息可帮助管理员识别需要更换但仍在使用的旧设备，以及未使用或未充分利用的昂贵设备。
    
- **技术支持**。 故障排除数据有助于的支持工程师确定用户无需收集服务器或客户端日志调用失败。 此信息可以很容易地访问和理解并没有深入的技术和知识的业务客户端的 Skype 业务服务器 2015年的 Skype 的技术支持人员。
    
- **系统故障排除**。使管理员能够检测可能阻止最终用户执行基本任务（如加入会议、建立呼叫或发送即时消息）的重大问题。
    
监控还提供了一种机制，允许 SIP 终结点 （如 Skype 业务) 提供管理员本来无法访问的故障排除信息：
  
- **影响质量的媒体指标**。这些指标处理呼叫本身的实际传输；它们提供了关于呼叫在网络中的传输过程的旅行日志。这些指标（包括数据包丢失、抖动和来回行程时间）提供了呼叫从离开某个人的终结点到呼叫到达其他人员的终结点期间所经历事情的相关信息。
    
- **报告给最终用户的问题**。 这些指标包括质量差通知 Skype，说话太轻而为业务提供给最终用户在哪里的麦克风，太远的情况下有一个较差的网络连接，或者是遇到质量较差，由于在另一个程序计算机正在使用的可用资源。
    
- **环境信息**。这些指标详细说明了呼叫质量因素，如所使用的麦克风和扬声器的类型、用户是否通过 VPN 连接进行连接以及用户是否使用无线连接。
    
在每个呼叫结束时，符合 SIP 的终结点会将该信息传输到实现该呼叫的前端服务器。你不必执行任何操作就能让终结点传输该信息；该行为内置于 SIP 协议中。但是，如果要收集和存储该信息，则需要安装和启用监控。如果确实安装并启用了监控，则呼叫信息将由前端服务器上运行的代理收集，并被中继到一对 SQL Server 数据库。监控服务（采用“统一数据收集代理”的形式）并置到所有前端服务器中。 
  
## <a name="define-your-requirements-for-monitoring"></a>定义监控要求
<a name="requirements"> </a>

仍有开始为安装和配置监视与 Skype 业务服务器 2015年之前应解决的几个关键问题：
  
 **想要何时安装监控？** 监视可以安装和配置在安装和配置业务服务器 2015; Skype 的同时Skype 业务服务器 2015年部署向导将为您提供在安装过程中将前端池与监视数据库相关联的机会。 或者，您可以安装监视安装 Skype 的业务服务器 2015 本身; 之后这可以通过使用拓扑生成器来与您的前端池和服务器监视的数据库，然后再将发布修订后的拓扑结构。
  
请记住，必须先安装并配置 SQL Server，然后再部署和配置监控。 但是，您只需部署 SQL Server，然后重试。将为您创建监视数据库，当您发布的业务服务器拓扑结构您 Skype。
  
 **要监视什么类型的数据？** Skype 的业务服务器 2015年使您能够监视两个一般类型的数据： 呼叫详细记录 (CDR) 数据和体验质量 (QoE) 数据。 呼叫详细记录提供了有助于您跟踪业务服务器 2015年功能，如语音 IP (VoIP) 电话联络; 通过 Skype 的使用情况即时消息 (IM);文件传输;音频/视频 (A / V) 会议;和应用程序共享会话。 此信息可帮助您知道哪些 Skype 业务服务器 2015年功能正在使用 （和它们的不是），还提供了有关何时使用这些功能的信息。 经验数据的质量使您可以维护质量的组织，包括诸如网络数据包丢失数中所做的音频和视频呼叫记录、 背景噪声和量"抖动"（数据包延迟的差异）。
  
如果您选择启用在 Skype 业务服务器 2015 可以启用监视的 CDR 和 QoE 监控监控，或者您可以选择启用一种类型的监视而禁用另一种类型。 例如，假设用户仅使用即时消息和文件传输，不进行音频或视频呼叫。 在这种情况下，可能没有理由启用 QoE 监控。 同样，业务服务器 2015年的 Skype 很容易来启用和禁用监视监视部署后。 例如，您可能选择部署监控但最初禁用 QoE 监控。 如果用户开始遇到音频或视频呼叫问题，您可以启用 QoE 监控并使用该数据帮助排查和解决这些问题

 **需要多少个后端监控数据库？** 它估计并入监视和存档数据库无法支持的业务服务器 2015年用户的 Skype 240,000）。 此外，一个监控数据库可以被多个前端池使用；如果组织中有三个前端池，则可以将全部三个池与同一后端存储关联。
  
对于许多组织来说，数据库容量并不是确定所需后端监控数据库数量的决定因素。相反，网络速度可能是更重要的考虑事项。假设你有三个前端池，但其中一个池位于慢速网络连接。在这种情况下，你可能想要使用两个监控数据库：一个数据库为网络连接良好的两个池提供服务，另一个数据库为网络连接较慢的池提供服务。
  
您还应考虑到有关业务服务器 2015 Skype 支持镜像数据库的使用。 “数据库镜像”提供了同时维护两个数据库副本，且每个数据库位于不同服务器上的方式。 每当将数据写入主数据库时，相同的数据也会写入镜像数据库。 如果主数据库故障或否则会变得不可用，您可以"故障转移"到镜像数据库通过使用简单的 Skype 业务服务器 2015 PowerShell 命令。 例如：
  
```
Invoke-CsDatabaseFailover -PoolFqdn atl-cs-001.litwareinc.com -DatabaseType "Monitoring" -NewPrincipal "Mirror"
```

这对于规划来说很重要，仅仅因为镜像需要将所需数据库数量翻倍：除了各个主数据库，还需要第二个数据库作为镜像。
  
 **您 Skype 业务服务器 2015年网站需要它们自己的自定义监视配置吗？** 您为业务服务器 2015年安装 Skype 时还安装全局集合的 CDR 和 QoE 配置设置;这些全局集合提供相同的 CDR 和 QoE 设置适用于整个组织的能力。 这在许多情况下将足够：比如通常您会想要为所有用户启用 CDR 监控。
  
但是，有时也可能当您想要将不同的设置应用到不同的站点。 例如，也许您希望使用 CDR 和 QoE 监控在雷德蒙得站点，但只能使用 CDR 在都柏林站点监视。 同样，您可能想要 60 天监视数据保留在雷德蒙的站点，但只需在都柏林站点 30 天内保持这种类型的数据。 Skype 的业务服务器 2015年使您可以在站点范围中; 创建独立的 CDR 和 QoE 配置设置的集合它使您能够以不同的方式管理每个站点。 （此过程包括两个启用和禁用监视配置管理设置，如多长时间的数据是要保留）。
  
请注意，您既可以在部署监控前也可以在部署监控后做出此决定。例如，您可以先部署监控，然后使用全局设置管理整个组织。如果以后改变主意，例如可以为 Redmond 站点创建单独的设置集合，然后使用这些设置管理 Redmond 的监控。（在站点作用域内应用的设置始终优先于在全局作用域内应用的设置。）如果您再次改变主意，只需删除应用于 Redmond 站点的配置设置。删除站点设置集合后，全局设置集合将自动应用于该站点。
  
## <a name="sql-requirements-for-monitoring"></a>监控的 SQL 要求
<a name="topologies"> </a>

当你启用监控时，会自动在每个前端服务器上安装并激活统一数据收集代理。你将需要安装下列数据库之一，并将其配置为用作监控数据的后端数据存储：
  
- Microsoft SQL Server 2008 R2 Enterprise Edition（64 位版本）
    
- Microsoft SQL Server 2008 R2 Standard Edition（64 位版本）
    
- Microsoft SQL Server 2012 Enterprise Edition（64 位版本）
    
- Microsoft SQL Server 2012 Standard Edition（64 位版本）
    
- Microsoft SQL Server 2014 Enterprise Edition（64 位版本）
    
- Microsoft SQL Server 2014 Standard Edition（64 位版本）
    
SQL Server 的 32 位版本无法用作后端存储。 业务服务器 2015年的 Skype 不支持 SQL Server 2012年的速成版的 SQL Server 2008年。 Skype 的业务服务器 2015年数据库要求的详细信息请参阅主题[数据库软件支持](http://technet.microsoft.com/library/e05d0032-bbea-4e61-987d-d07b1c045fd5.aspx)。
  
监控数据可以与其他类型的数据共享 SQL Server 实例。通常，呼叫详细信息记录数据库 (LcsCdr) 和体验质量数据库 (QoEMetrics) 共享相同的 SQL 实例；这两个监控数据库与存档数据库 (LcsLog) 位于相同的 SQL 实例中也是很常见的。对 SQL Server 实例的唯一真正要求是，SQL Server 的任何一个实例仅限于以下各项：
  
- Skype 业务服务器 2015年后端数据库的一个实例。 （作为一般规则，不建议搭配监视数据库中的同一个 SQL 实例，或甚至在同一计算机上，作为后端数据库。 尽管技术上是可行的运行监视数据库的后端数据库所需的磁盘空间使用的风险。） 
    
- 呼叫详细信息记录数据库的一个实例。
    
- 体验质量数据库的一个实例。
    
- 存档数据库的一个实例。
    
换句话说，你不能在同一个 SQL Server 实例中拥有两个 LcsCdr 数据库实例。如果你需要多个 LcsCdr 数据库实例，则需要配置多个 SQL Server 实例。

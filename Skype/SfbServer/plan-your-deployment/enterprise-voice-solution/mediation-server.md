# <a name="mediation-server-component-in-skype-for-business-server-2015"></a>Skype for Business Server 2015 中的中介服务器组件
 
了解在 Skype 的中介服务器业务服务器，包括其受支持的拓扑和 M:N 中继、 媒体回避和调用许可控制及其关系。
  
若要部署企业语音，您必须部署一个或多个中介服务器。 
  
中介服务器转换内部企业语音基础结构和公用交换的电话网络 (PSTN) 网关或会话启动协议 (SIP) 干线之间的信号传输。 在某些配置下，还会在这些点之间转换媒体本身。
  
在业务服务器端的 Skype，中介服务器监听一个共同的 TLS (MTLS) 在传输地址上。 在网关端，中介服务器侦听所有关联的侦听端口关联的中继。 所有合格网关必须支持 TLS，但也可以启用 TCP。 不支持 TLS 的网关将支持 TCP。
  
如果您的环境中还有现有的公用分支交换 (PBX)，中介服务器处理企业语音用户和 PBX 之间的电话。 如果您 PBX IP PBX，可创建之间的 PBX 和中介服务器的 SIP 直接连接。 如果您 PBX 时分复用 (TDM) PBX，您还必须部署中介服务器和 PBX 之间的 PSTN 网关。
  
默认情况下，中介服务器搭配使用前端服务器。 此外可以在一个独立的池部署中介服务器。
  
## <a name="what-mediation-server-does"></a>中介服务器的功能

中介服务器的主要功能如下所示：
  
- 加密和解密 srtp 结合业务服务器端为 Skype 上。 
    
- 将 TCP 上的 SIP（针对不支持 TLS 的网关）转换为相互 TLS 上的 SIP。
    
- 正在转换媒体流业务服务器的 Skype 之间的中介服务器的网关等。
    
- 将网络外部的客户端连接到内部 ICE 组件，使媒体可以遍历 NAT 和防火墙。
    
- 网关不支持，例如，在企业语音 clien.t 上的远程工作人员从调用的调用流的媒介作用
    
- 在包含 SIP 中继的部署中，配合 SIP 中继服务提供商提供 PSTN 支持，从而不再需要 PSTN 网关。

## <a name="mn-trunk"></a>M:N 中继

Skype 业务服务器的呼叫路由目的的干线的定义中的支持的灵活性。 主干是具有网关并侦听的端口号中介服务器的侦听端口号，之间的逻辑关联。 这意味着几件事： 中介服务器可以有多个中继到相同的网关;中介服务器可以有多个中继到其他网关;相反一个网关可以有多个不同的中介服务器的中继。
  
添加到您的业务拓扑使用拓扑生成器的 Skype 的网关时，仍必须创建根主干。 可以处理给定的中介服务器的网关的数目在繁忙高峰取决于服务器的处理能力。 如果您部署中介服务器满足 Skype 业务服务器的最低硬件要求的硬件上，所述[业务服务器 2015年的 Skype 的服务要求](../../plan-your-deployment/requirements-for-your-environment/server-requirements.md)，可以大约处理独立的中介服务器1000 的电话。 中介服务器执行转换，但仍传送调用多个网关，即使网关不支持媒体回避。
  
在定义的呼叫路由时，您指定与该路由关联的中继，但未指定与该路由关联的中介服务器。 相反，您可以使用拓扑生成器将中继与中介服务器相关联。 换句话说，路由确定要用于调用的干线，随后，该干线与关联的中介服务器发送该调用的信号。
  
中介服务器可以部署为一个池;该池可以搭配与前端池，或者它可以作为一个独立的池部署。 当中介服务器搭配与前端池时，池的大小最多只能包含 12 （注册池的大小限制）。 集合在一起，这些功能提高了可靠性并部署灵活性对于中介服务器，但它们需要在以下类似的功能：
  
- **PSTN 网关。** Skype 业务服务器限定网关必须实现 DNS 负载平衡，从而可以跨池充当中介服务器的从而为负载平衡调用一个池的负载平衡器的限定公共交换的电话网络 (PSTN) 网关.
    
- **会话边界控制器。** 对于 SIP 中继，对等实体是 Internet 电话服务提供商的会话边界控制器 (SBC)。 对 SBC 的方向从中介服务器池，SBC 可以接收来自池中任何中介服务器的连接。 从 SBC 池的方向，流量可以发给池中任何中介服务器。 实现此功能的一个方法是通过 DNS 负载平衡（如果服务提供商和 SBC 支持）。 另一种方法是为池中的所有中介服务器的 IP 地址的服务提供商和服务提供商将其 SBC 在作为单独的 SIP 中继为配置每个中介服务器。 然后，服务提供商将处理自己的服务器的负载平衡。 并非所有服务提供商或 SBC 都可以支持这些功能。 此外，服务提供商可能会对此功能收取额外费用。 通常，每个到 SBC 的 SIP 中继按月收费。
    
- **IP-PBX。** 在 SIP IP PBX 中止从中介服务器池方向，IP PBX 可以接收来自池中任何中介服务器的连接。 从 IP PBX 池的方向，流量可以发给池中任何中介服务器。 由于大多数 IP Pbx 不支持 DNS 负载平衡，因此建议个别直接 SIP 连接从 IP PBX 定义为池中的每个中介服务器。 然后，IP-PBX 将通过向中继组分发流量处理自己的负载平衡。 假定中继组在 IP-PBX 中具有一组一致的路由规则。 是否某个特定 IP PBX 支持此中继组概念以及如何与 IP-PBX 的自己冗余相交，它和群集体系结构需要确定之前，您可以决定是否中介服务器群集可以与 IP PBX 正确交互。
    
中介服务器池必须具有与它交互的对等网关的统一视图。 这意味着池中的所有成员都能从配置存储访问对等网关的同一定义，并且都可能针对传出呼叫与其进行交互。 因此，是没有办法分段池，以便与只拨出呼叫特定网关等的一些中介服务器通信。 如果需要此类分段，则必须使用单独的中介服务器池。 例如，如果与池进行交互的 PSTN 网关、SIP 中继或 IP-PBX 中的相关功能（如本主题中前面所述）不存在，则此情况适用。
  
特定的 PSTN 网关，IP PBX 或 SIP 中继等可路由到多个中介服务器或中继。 网关，可以控制特定池的中介服务器的数目取决于使用媒体回避的调用次数。 如果大量的调用使用媒体回避，池中中介服务器可以处理许多发出更多调用，因为只有信号传输层处理是必要的。 
  
## <a name="call-admission-control-and-mediation-server"></a>呼叫允许控制和中介服务器

呼叫允许控制 (CAC) 根据可用带宽管理实时会话建立，以帮助改善拥堵网络中的用户体验质量 (QoE)。 为了支持这一点，中介服务器负责的两个交互业务服务器端为 Skype 和网关侧上的带宽管理。 在呼叫允许控制中，呼叫的端接实体处理带宽预留。 中介服务器的网关一端与交互网关等 （PSTN 网关，IP PBX、 SBC） 不支持 Skype 业务服务器调用许可控制。 因此，中介服务器能够处理带宽交互代表其网关等。 只要有可能，中介服务器将提前预留带宽。 如果不可能（例如，如果网关端的最终媒体终结点位置对于发往对等网关的传出呼叫是未知的），则会在发出呼叫时预留带宽。 此行为可能导致带宽订阅过度，但这是防止错误响铃的唯一方法。
  
媒体旁路和带宽预留相互排斥。 如果对呼叫使用媒体旁路功能，则不会对该呼叫执行呼叫允许控制。 此处假定前提是此次呼叫不涉及具有限定带宽的链接。 如果调用许可控制用于涉及中介服务器的特定调用，该调用不能利用媒体回避。
  
有关媒体的详细信息绕过或调用许可控制，请参阅[规划媒体业务 2015年的 Skype 在绕过](media-bypass.md)或[计划中的业务服务器 2015年的 Skype 通话许可控制](call-admission-control.md)。
  
## <a name="enhanced-9-1-1-e9-1-1-and-mediation-server"></a>增强型 9-1-1 (E9-1-1) 和中介服务器

中介服务器的扩展的功能，以便可以正确处理增强 9-1-1 (E9-1-1) 服务提供商。 中介服务器不需要任何特殊的配置。 默认情况下，包括在其交互，而与网关对方 （PSTN 网关，IP PBX 或互联网电话服务提供程序，包括 E9-1-1 服务的 SBC 的中介服务器的 SIP 协议 SIP 扩展所需的 E9-1-1 的交互是提供程序）
  
SIP 中继到 E9-1-1 的服务提供商可以在现有的中介服务器池终止还是就需要独立的中介服务器将取决于是否 E9-1-1 SBC 可以与进行交互的中介服务器池。 有关详细信息，请参阅[在业务服务器 2015年的 Skype 的 M:N 干线](m-n-trunk.md)。
  
## <a name="media-bypass-and-mediation-server"></a>媒体旁路和中介服务器

媒体旁路是 Skype 业务服务器功能，使管理员能够配置呼叫路由不遍历中介服务器的情况下直接用户终结点和公用交换的电话网络 (PSTN) 网关之间流动。 媒体回避通过减少延迟、 不必要的翻译、 数据包丢失的可能性和潜在故障点的呼叫质量有所改善。 没有中介服务器的远程站点由一个或多个有限带宽的 WAN 链接连接到中心站点，其中媒体回避通过启用媒体从远程站点的客户端流动到其本地网关，而直接降低了带宽要求第一次无需流经 WAN 链接到中央站点的中介服务器的来回。这种减少在媒体处理还补充了控制多个网关中介服务器的能力。
  
媒体旁路功能和呼叫允许控制 (CAC) 相互排斥。如果某次呼叫使用媒体旁路功能，则不会为此次呼叫执行 CAC。假定前提是此次呼叫不涉及具有限定带宽的链接。
  
## <a name="topologies-for-mediation-server"></a>中介服务器拓扑

Skype 业务服务器，中介服务器是默认搭配使用标准版服务器、 前端池或高存活力的分支装置。 在前端池中的所有中介服务器必须都配置都完全相同。
  
如果性能是一个问题，可能会部署在专用的独立池中的一个或多个中介服务器版本号可取的做法。 部署 SIP 中继时，强烈推荐采用独立池。 
  
如果将直接 SIP 连接部署到支持媒体回避限定 PSTN 网关和 DNS 负载平衡、 独立的中介服务器池则不需要。 这是因为 DNS 负载平衡到中介服务器池能够限定网关并能够从池中任何中介服务器接收通信。
  
我们还建议，您布置中介服务器前端池上时您已部署 IP Pbx 或连接到互联网电话服务服务器提供程序的会话边框控制器 (SBC)，只要满足以下条件的任何操作：
  
- IP PBX 或 SBC 被配置为从池中任何中介服务器接收通信，可以将路由通信统一到池中的所有中介服务器。
    
- IP PBX 不支持媒体迂回，但承载中介服务器的前端池可以处理语音转码媒体回避不到适用的电话。
    
您可以使用 Microsoft Lync Server 2013，规划工具来评估是否想要布置中介服务器的前端池可以处理的负载。 如果您的环境不能满足这些要求，您必须部署独立的中介服务器池。

## <a name="planning-decisions-for-mediation-server"></a>规划中介服务器决策

本主题介绍中介服务器部署，您需要规划决策
  
### <a name="collocated-or-stand-alone-mediation-server"></a>按顺序排列或独立的中介服务器？

中介服务器是默认搭配使用标准版服务器或在中心站点的前端池在前端服务器上。 可以处理的公用电话交换网 (PSTN) 呼叫数和池中所需的计算机数取决于以下因素：
  
- 网关中介服务器池控制的对等方的数量
    
- 通过这些网关的高流量时段
    
- 其媒体绕过中介服务器的调用的调用的百分比
    
规划时，请确保考虑未进行媒体旁路配置的 PSTN 呼叫和 A/V 会议的媒体处理要求，同时考虑处理必须支持的忙时呼叫数量的信号交互所需的处理要求。 如果没有足够的 CPU，那么必须部署独立的中介服务器; 池并且需要 PSTN 网关，IP 的 Pbx 和 SBCs 拆分成将由在一个池中的并入的中介服务器和独立的中介服务器中一个或多个独立的池控制的子集。
  
如果您部署了 PSTN 网关，IP Pbx 的或会话边框控制器 (SBCs) 不支持正确的功能进行交互的中介服务器，包括以下内容，然后他们将需要将与独立的池组成的池一台的中介服务器：
  
- 执行域名系统 (DNS) 负载平衡在池中的中介服务器之间的网络层 （或否则路由通信统一到池中的所有中介服务器）
    
- 接受从池中任何中介服务器的通信
    
您可以使用 Microsoft Lync Server 2013，规划工具来评估是否配置前端池与中介服务器可以处理的负载。 如果您的环境不能满足这些要求，您必须部署独立的中介服务器池。
  
### <a name="central-site-and-branch-site-considerations"></a>中央站点和分支站点注意事项

 场地中央的中介服务器可用于路由呼叫到 IP Pbx 或 PSTN 网关处的分支站点。 如果部署 SIP 中继，但是，必须部署中介服务器站点每个中继的终止位置。 有中介服务器在中心站点路由呼叫 IP PBX 或 PSTN 网关在分支站点不需要绕过使用媒体。 但是，如果可以启用媒体旁路，则可以降低媒体路径延迟，从而改善媒体质量，因为媒体路径不再需要遵循信号路径。 媒体回避也减少了池上的处理负载。
  
> [!NOTE]
> 媒体旁路将不会与每个 PSTN 网关、IP-PBX 和 SBC 进行交互操作。 Microsoft 与认证合作伙伴一起对一组 PSTN 网关和 SBC 进行了测试，另外也对 Cisco IP-PBX 进行了一些测试。 绕过媒体仅支持产品和版本中列出[统一通信开放互操作性计划-Lync Server](https://go.microsoft.com/fwlink/p/?LinkId=268730)。 
  
如果需要分支站点恢复能力，高存活力的分支装置或前端服务器、 中介服务器和网关的组合必须部署在分支站点。 （分支站点恢复能力与假设是存在和会议不适应性在站点。）有关分支站点规划的语音指导，请参阅[规划业务服务器 2015年的 Skype 企业语音恢复功能](enterprise-voice-resiliency.md)。
  
对于 IP PBX 与交互，如果 IP PBX 无法正常支持与多个早期对话的早期媒体交互和 RFC 3960 交互，可能有的第一个剪辑的传入调用从 IP PBX Skype 的问候几句业务的终结点。 此问题可能更严重，如果中央站点的中介服务器路由的 IP PBX 电话路由分支地点，将终止，因为信号来完成需要更多的时间。 如果您遇到这种情况，部署中介服务器在分部地点是第的唯一的方法，以减少修剪几个字。
  
最后，如果中心站点 TDM PBX，或者您 IP PBX 并不能排除需要 PSTN 网关，则必须部署上连接中介服务器和 PBX 的呼叫路由的网关。
  
> [!NOTE]
> 要提高独立的中介服务器的媒体性能，应在这些服务器的网络适配器上启用接收方缩放 (RSS)。 通过启用 RSS，服务器上的多个处理器能够以并行方式处理传入数据包。 有关详细信息，请参阅["接收端扩展增强功能在 Windows 服务器"](https://go.microsoft.com/fwlink/p/?LinkId=268731)。 有关如何启用 RSS 的详细信息，请参阅网络适配器文档。

<span data-ttu-id="1b0d7-212">在 CQD 中，当 ICE 连接失败或者没有报告计算流分类所需的所有指标时，流标记为未经分类。</span><span class="sxs-lookup"><span data-stu-id="1b0d7-212">In CQD, a stream is marked unclassified when ICE connectivity fails or when all the metrics required to compute the stream classification are not reported.</span></span>

<span data-ttu-id="1b0d7-213">要检查 ICE 连接故障，请检查 "First Connectivity Ice" 和 "Second Connection Ice" 维度是否为 "FAILED" 值。</span><span class="sxs-lookup"><span data-stu-id="1b0d7-213">To check for ICE connectivity failures, examine the dimensions "First Connectivity Ice" and "Second Connectivity Ice" for a "FAILED" value.</span></span> <span data-ttu-id="1b0d7-214">如果任一值指示失败，则该流将被标记为未经分类。</span><span class="sxs-lookup"><span data-stu-id="1b0d7-214">If either value indicates a failure, the stream will be marked as unclassified.</span></span>

<span data-ttu-id="1b0d7-215">如果未经分类的流成功执行 ICE 连接，则该流可能被视为未经分类，因为未报告关键流指标。</span><span class="sxs-lookup"><span data-stu-id="1b0d7-215">If ICE connectivity succeeded for an unclassified stream, the stream is likely considered unclassified because key stream metrics were not reported.</span></span> <span data-ttu-id="1b0d7-216">这些指标可能不会被报告的原因有几个：</span><span class="sxs-lookup"><span data-stu-id="1b0d7-216">There are a few reasons these metrics may not be reported:</span></span>

- <span data-ttu-id="1b0d7-217">**没有收到 QoE 报告** - 用于分类的指标在呼叫结束时发送的 QoE 报告中予以报告。</span><span class="sxs-lookup"><span data-stu-id="1b0d7-217">**QoE reports were not received** - The metrics used for classification are reported in a QoE report sent at the end of a call.</span></span> <span data-ttu-id="1b0d7-218">如果未生成此报告（例如，由于某些第三方端点可能不发送 QoE）或无法发送（例如，由于网络中断），CQD 无法对流进行分类。</span><span class="sxs-lookup"><span data-stu-id="1b0d7-218">If this report is not produced (e.g., because some third-party endpoints may not send QoE) or was not able to be sent (e.g., because of a network outage), CQD is unable to classify the stream.</span></span>

> [!TIP]
> <span data-ttu-id="1b0d7-219">"QoE Record Available" 维度可用于确定是否收到某个流的 QoE 报告。</span><span class="sxs-lookup"><span data-stu-id="1b0d7-219">The "QoE Record Available" dimension can be used to determine whether a QoE report was received for a stream.</span></span> <span data-ttu-id="1b0d7-220">请注意，如果从任一端点接收到 QoE 报告，则此维度的值将为 "True"。</span><span class="sxs-lookup"><span data-stu-id="1b0d7-220">Note that this dimension will have a value of "True" if a QoE report was received from either endpoint.</span></span> <span data-ttu-id="1b0d7-221">为了最准确地报告指标，需要两个端点的 QoE 报告。</span><span class="sxs-lookup"><span data-stu-id="1b0d7-221">A QoE report from both endpoints is required for the most accurate reporting of metrics.</span></span>

- <span data-ttu-id="1b0d7-222">**短时间呼叫** - 短时间呼叫可能没有足够的媒体活动来计算关键流指标。</span><span class="sxs-lookup"><span data-stu-id="1b0d7-222">**Short calls** - Short calls may not have enough media activity to compute key stream metrics.</span></span> <span data-ttu-id="1b0d7-223">如果没有这些指标，CQD 无法对流进行分类。</span><span class="sxs-lookup"><span data-stu-id="1b0d7-223">Without these metrics, CQD is unable to classify the stream.</span></span>

> [!TIP]
> <span data-ttu-id="1b0d7-224">可以使用 "Duration (Seconds)"、"Duration (Minutes)"、"Duration 5 seconds or less" 和 "Duration 60 seconds or more" 等维度来确定流的持续时间。</span><span class="sxs-lookup"><span data-stu-id="1b0d7-224">The dimensions "Duration (Seconds)", "Duration (Minutes)", "Duration 5 seconds or less", and "Duration 60 seconds or more" can be used to determine the duration of a stream.</span></span> <span data-ttu-id="1b0d7-225">"Avg Call Duration" 度量也可用于计算一组流的平均持续时间。</span><span class="sxs-lookup"><span data-stu-id="1b0d7-225">The measurement "Avg Call Duration" can also be used to compute the average duration for a set of streams.</span></span>

- <span data-ttu-id="1b0d7-226">**低数据包利用率** - 就像“短时间呼叫”情况那样，关键流指标的计算需要足够的数据包利用率。</span><span class="sxs-lookup"><span data-stu-id="1b0d7-226">**Low packet utilization** - Like the "short call" scenario, sufficient packet utilization is required for computation of key stream metrics.</span></span> <span data-ttu-id="1b0d7-227">如果没有这些指标，CQD 无法对流进行分类。</span><span class="sxs-lookup"><span data-stu-id="1b0d7-227">Without these metrics, CQD is unable to classify the stream.</span></span>
    - <span data-ttu-id="1b0d7-228">当用户加入会议来聆听演讲者但从不说话（可能像在大多数呼叫中那样将麦克风静音）时，会出现常见的低数据包利用率。</span><span class="sxs-lookup"><span data-stu-id="1b0d7-228">A common low packet utilization scenario occurs when a user joins a meeting to listen to the presenter but never speaks (likely muting the microphone for most of the call).</span></span> <span data-ttu-id="1b0d7-229">在这种情况下，一个音频流将具有高数据包利用率（入站到客户端），而另一个音频流几乎不具有数据包利用率（从客户端出站）。</span><span class="sxs-lookup"><span data-stu-id="1b0d7-229">In such a scenario, one audio stream will have high packet utilization (inbound to the client) while the other will have little to no packet utilization (outbound from the client).</span></span> <span data-ttu-id="1b0d7-230">在这种情况下，流的持续时间可能为一小时或更长，但由于麦克风被静音，从客户端到服务器的流数据包利用率将非常低，从而导致未分类的流。</span><span class="sxs-lookup"><span data-stu-id="1b0d7-230">In this scenario, the duration of the stream may be an hour or longer but the packet utilization on the stream from the client to the server will be extremely low due to the microphone being muted, resulting in an unclassified stream.</span></span>
